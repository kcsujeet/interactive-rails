## Batch 3: L30 static-cache-key invalidation fix

### Problem

L30's step 4 had the player pick `Rails.cache.fetch("trending_products",
expires_in: 5.minutes) { ... }` as the correct answer, with a literal
string key. Step 5 then taught `belongs_to :product, touch: true` for
"cache invalidation" — but a literal-string cache key never reads
`updated_at`, so `touch: true` did nothing. Step 5's description even
said "cache keys that depend on updated_at are invalidated," promising
something step 4 didn't deliver. The Review model comment in the code
preview (`# Cache keys that include updated_at are automatically
invalidated`) was aspirational fiction; no key in the level included
`updated_at`.

### Decision

Add a versioned-key option as the new correct answer:
`Rails.cache.fetch([Product.maximum(:updated_at).to_i, "trending_products"], expires_in: 5.minutes)`.
The integer max-updated-at means any `touch` on a Product bumps the key,
and the next request gets a different key (cache miss → recompute).
Demoted the old static-key-with-expires-in answer to a 4th wrong option
that explicitly teaches why static keys can't reflect record changes.

### Changes

- **L30 Caching** (`Level30Caching.tsx`)
  - `CACHE_FETCH_OPTIONS`: 3 wrong + 1 correct (was 2 wrong + 1 correct).
    New wrong option `wrong-static-key` is the previous "correct" answer,
    now flagged with feedback explaining the static-key invalidation gap.
    New correct answer uses `[Product.maximum(:updated_at).to_i, "trending_products"]`
    as the key.
  - Step 4 description: now mentions "the cache must respect both
    expiration AND changes to the underlying records, so a fresh review
    is reflected immediately rather than waiting for the timer."
  - Code preview at the resolved state: cache key updated to match the
    new correct answer.
- **L30 tests** (`Level30Caching.test.ts`): mirrored the new
  `CACHE_FETCH_OPTIONS` (4 entries) so the test data matches the

## Batch 8: L14 `params.expect` failure-mode demo

### Problem

L14 taught the syntax of `params.expect(product: [:name, :description])`
but never demonstrated *why* it is preferred over the older
`params.require(:product).permit(...)` pattern. The whole point of
`expect` is its strictness: it enforces the **shape** of the request,
not just the key whitelist. The level had no concrete demonstration of
this property.

### Decision

Two additions:

1. A new stress-test scenario in the reward phase that demonstrates
   the type-shape enforcement: an attacker sends `product=hacked` (a
   plain string) instead of `product[name]=...` (a hash with sub-keys).
   `params.expect(product: [:name, :description])` raises
   `ActionController::ParameterMissing` and Rails returns
   `400 Bad Request`. This shows the strictness in action without
   needing an extra observe-phase probe.
2. A new bullet section in `learningContent.conceptExplanation`
   explaining the why in plain language: shape vs keys, the explicit
   400 vs the subtler bad-data-in-database failure mode of `permit`,
   and the rule "use `expect` for new Rails 8 code, it fails loudly at
   the security boundary."

### Changes

- **L14 component** (`Level14StrongParams.tsx`)
  - Added `'malformed-payload'` to `STRESS_SCENARIOS`. Description:
    *"Attacker sends product as a string ("hacked") instead of a hash.

    Method POST, expectedResult `blocked`.
- **L14 content** (`level-14-strong-params.ts`)

## Batch 10: L35 Active Storage URL-generation clarification

### Audit

The audit (and CLAUDE.md item 5) flagged that
`create_before_direct_upload!` writes a local `Blob` row and computes
the presigned URL from credentials in `config/storage.yml`. There is
no network call to S3 during URL generation. A previous version of

  clarification of how Rails produces the URL. A first-time reader
  could easily assume Rails goes to S3 to fetch one.

### Decision

`conceptExplanation`: add an explicit "How a presigned URL actually
gets made" section that pre-empts the misconception, and rewrite the
six-step architecture list with concrete HTTP verbs and clear
"no file bytes yet" callouts.

### Changes

- **L35 content** (`level-35-active-storage.ts`)

     scope. Future audits should not propose demoting them. Service
     Objects + Result + Dry::Validation + Query Objects + Pundit are
     all *correct* choices for billion-dollar SaaS architecture.
2. **A Hotwire act is NOT what's missing.**
   - Original critique: "Rails IS its view layer, add Hotwire."
   - User's pushback: a billion-dollar SaaS dashboard typically uses

### Refined top three (staff-engineer)

## Batch 17: L11 callback content rewrite + enum step (Batch B)

### Problem

L11 had been teaching the **callback spaghetti anti-pattern as the
correct answer**: step 1's "right" choice was `after_create
:send_welcome_email` and step 3's was `after_commit` for "external
API calls." Both directly contradict `rails-principles.md`
("NEVER in callbacks: emails, API calls, job enqueues") and
`rails-conventions.md` ("Callbacks: normalization only"). Adding
the missing enum step on top of the anti-pattern teaching would
have been cosmetic. Foundation gap #4 (enum) was also still open.

### Decision

Rework all four build steps so the level teaches the curriculum's
actual position on callbacks:

| Step | Was | Now |
| --- | --- | --- |
| 0 | Choose Normalization (`normalizes`) | Normalize the Email — same correct answer; tightened wrong-option feedback |
| 1 | Add Callback (correct: `after_create :send_welcome_email`) | **Add a Status Enum** — string-encoded `enum :status, draft: "draft", listed: "listed", sold: "sold"` |
| 2 | Order Callbacks (lifecycle trivia) | **Send the Welcome Email** — correct: `UserMailer.welcome(@user).deliver_later` from `UsersController#create`; both callback options are wrong |
| 3 | Avoid Pitfall (correct: `after_commit` for API calls) | **Sync to External Service** — correct: `AccountingSyncJob.perform_later(product.id)` from the controller; both callback options are wrong |

The lifecycle-ordering trivia step (Order Callbacks) was dropped:
in a level whose new message is "callbacks for normalization only,
side effects belong elsewhere," memorising the order in which all
callbacks fire is misaligned. Enum takes its slot.

### Encoding choice for the enum

String-encoded, per `rails-conventions.md`. Integer-encoded is the
explicit wrong option, with feedback explaining the production
gotchas (opaque DB dumps, dangerous reorder semantics, unreadable
SQL). The "plain string column without enum" wrong option teaches
why a fixed set deserves first-class model support.

### Anti-pattern motivator framing

Steps 2 and 3 are now structured as anti-pattern motivators:

- The two callback-based wrong options (`after_create`,
  `after_commit`) demonstrate the real failure modes (test runs
  fire real mail, seeds fire real mail, trigger invisible to the
  controller, save blocks on third-party vendor uptime).
- The correct option puts the side effect where the controller (or
  a service the controller calls) explicitly invokes it.

This matches `rails-principles.md`'s "Explicit over implicit" and
"Side effects in callbacks make the model untestable" rules.

- `data/build-steps.ts` — full rewrite: new STEP_DEFS,
  STATUS_ENUM_OPTIONS, WELCOME_EMAIL_OPTIONS, EXTERNAL_SYNC_OPTIONS;
  NORMALIZATION_OPTIONS feedback tightened.

  artifact-accumulation model: User model normalized after step 0,

## Batch 18: L30 cache stampede (race_condition_ttl)

### Problem

`rails-conventions.md` already calls out cache stampede prevention
as a production-safe default ("on hot keys, pair `expires_in` with
`race_condition_ttl: 10.seconds`"), but L30 was teaching a versioned
key + `expires_in` only. Under heavy traffic, the moment a hot
cache key expires, every concurrent request sees a miss
simultaneously and they all hammer the database with parallel
recomputations. The level taught the cache pattern but stopped
short of the at-scale gotcha.

### Decision

Add the stampede gotcha to the existing "Add Cache Fetch" step
rather than creating a new step. Step 4 had four options (read+write,
no-expires, static-key, versioned-key-correct). Now five:

1. `read + manual write` (wrong) — kept
2. `fetch with no expires_in` (wrong) — kept
3. `fetch with static key + expires_in` (wrong) — kept
4. **NEW**: `fetch with versioned key + expires_in` (wrong, missing
   stampede protection) — feedback explains thundering herd
5. `fetch with versioned key + expires_in + race_condition_ttl: 10.seconds`
   (correct)

The new wrong option ("right shape, missing one parameter") forces
the player to recognize that "versioned key + expires_in" is
*almost* there. Feedback explains the failure mode without naming
the missing parameter.

Step description updated to motivate the new constraint without
revealing the answer:
> "Two constraints: (1) the cache must respect both expiration AND
>  changes to the underlying records, so a fresh review is reflected
>  immediately, and (2) under heavy traffic, when the key expires
>  only ONE concurrent request should recompute, while the rest
>  keep serving the slightly-stale value."

### Files changed

- `Level30Caching.tsx`
  - `CACHE_FETCH_OPTIONS`: 5 options, new `wrong-no-stampede` slot.
  - `OPTION_STEP_CONFIG[4].description`: rewritten with the new

  - `getCodeFiles` step-5 service rendering: `Rails.cache.fetch`

- `content/level-30-caching.ts`

  - `railsCodeExample`: service block now uses the production-safe

  - `commonMistakes`: new entry on cache stampede.

## Batch 19: L32 add delegated_type contrast

### Problem

L32 teaches `polymorphic: true` but contrasts it only against STI
in passing ("when types need different columns, use STI or
separate tables"). It never mentions `delegated_type` (Rails 6.1+),
which is specifically the modern alternative when STI breaks down.

`rails-principles.md` already calls this out:

> STI is a trap for "inherited entities that share 80% of columns"
> — when subtype-specific columns dominate, polymorphic
> associations or `delegated_type` are the right answer.

The level was missing the third leg of the design space.

### Decision

Add a "Three patterns, one design space" section to the
`conceptExplanation` that maps the three Rails macros to their
problem shapes:

- **`polymorphic: true`** (this level): one child, many parent
  types (uniform children, varied parents).
- **STI**: one parent, many subtypes that share 90%+ of columns.
- **`delegated_type`**: one parent, many subtypes with their own
  columns. The Rails 6.1+ fix for "STI is wrong here."

Includes a short canonical code snippet for `delegated_type` and a
"quick decision" three-line summary at the bottom so a reader can
pick the right tool without re-reading the whole section.

`whenToUse` updated with a one-sentence pointer to `delegated_type`
for the opposite-shape case. Added the
`ActiveRecord::DelegatedType` API doc to `furtherReading`.
