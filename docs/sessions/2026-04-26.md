## Batch 3: L30 static-cache-key invalidation fix

### Problem

L30's step 4 had the player pick `Rails.cache.fetch("trending_products",
expires_in: 5.minutes) { ... }` as the correct answer, with a literal
string key. Step 5 then taught `belongs_to :product, touch: true` for
"cache invalidation" — but a literal-string cache key never reads
`updated_at`, so `touch: true` did nothing. Step 5's description even
said "cache keys that depend on updated_at are invalidated," promising
something step 4 didn't deliver. The Review model comment in the code
preview (`# Cache keys that include updated_at are automatically
invalidated`) was aspirational fiction; no key in the level included
`updated_at`.

### Decision

Add a versioned-key option as the new correct answer:
`Rails.cache.fetch([Product.maximum(:updated_at).to_i, "trending_products"], expires_in: 5.minutes)`.
The integer max-updated-at means any `touch` on a Product bumps the key,
and the next request gets a different key (cache miss → recompute).
Demoted the old static-key-with-expires-in answer to a 4th wrong option
that explicitly teaches why static keys can't reflect record changes.

### Changes

- **L30 Caching** (`Level30Caching.tsx`)
  - `CACHE_FETCH_OPTIONS`: 3 wrong + 1 correct (was 2 wrong + 1 correct).
    New wrong option `wrong-static-key` is the previous "correct" answer,
    now flagged with feedback explaining the static-key invalidation gap.
    New correct answer uses `[Product.maximum(:updated_at).to_i, "trending_products"]`
    as the key.
  - Step 4 description: now mentions "the cache must respect both
    expiration AND changes to the underlying records, so a fresh review
    is reflected immediately rather than waiting for the timer."
  - Code preview at the resolved state: cache key updated to match the
    new correct answer.
- **L30 tests** (`Level30Caching.test.ts`): mirrored the new
  `CACHE_FETCH_OPTIONS` (4 entries) so the test data matches the
  component data.

### Out of scope

## Batch 4: L24 `includes` vs `eager_load` reframe

### Problem

Step 2 ("Fix Filtered Query") made `Product.eager_load(:tags).where(tags:
{ active: true })` the correct answer and demoted
`Product.includes(:tags).where(...)` to a wrong option, with feedback
that called `includes` "implicit" — framing implicitness as a flaw. The
problem: `includes` auto-promoting to `LEFT OUTER JOIN` when it detects
a WHERE on the joined table is precisely its job. It is the idiomatic
Rails behavior. Marking it wrong taught cargo-cult verbosity.

### Decision

Flip: `includes` is correct (the Rails idiom). `eager_load` is demoted
to a wrong option whose feedback explains it produces identical SQL
(Rails auto-promotes anyway), so picking the more verbose verb here
adds words without changing behavior. `preload` stays wrong (Rails
raises on filter-across-separate-queries).

### Changes

- **L24 component** (`Level24EagerLoading.tsx:662-687`)
  - Step 2 description: removed "explicit control and best performance"
    framing; now says "pick the method that handles this with the least
    ceremony — idiomatic Rails should do the right thing automatically."
  - `eager-load` option: was correct, now wrong with new feedback.
  - `includes-where` option: was wrong, now correct (no feedback).
  - Order preserved (preload, eager-load, includes-where), so correct
    answer is at index 2 (not first) — passes the "correct never first"
    check.
- **L24 test** (`Level24EagerLoading.test.ts`)
  - Mirrored `OPTION_STEP_2` data.
  - Updated the assertion `step 2 correct answer uses eager_load for
    filtering` to `step 2 correct answer uses includes for filtering
    (Rails auto-promotes to JOIN)`. Added explicit `not.toContain('eager_load')`
    so the test catches a regression in either direction.
