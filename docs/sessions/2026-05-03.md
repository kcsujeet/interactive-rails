## Codebase validator: design + build

After yesterday's Act 1 ground-truth pass shipped, evaluated three approaches for keeping level snapshots in sync with real myapp going forward:

1. **Per-level git-tag snapshots → JSON manifest → React consumer** (spike). Built end-to-end yesterday. Works, but 3.5MB manifest at scale, requires refactoring `getCodeFiles` reveal logic, and per-step granularity becomes its own problem.
2. **Continue manual ground-truth audits per act**. Brittle and high-touch.
3. **Validator script that compares hand-authored level snapshots against real myapp at git tags, reports drift**. Levels stay hand-authored and pedagogically tuned; correctness is enforced by a separate, runnable check.

User picked (3). Spike scaffolding from yesterday's experiment was reverted in commit a97dbcc to keep the slate clean.

### Design (docs/codebase-validator.md)

- **Scope (v1)**: validate only `registerLevelCode`'s output (the codebase viewer's snapshot). Per-step `getCodeFiles` content is a superset; deferred.
- **Mapping**: `act{X}-level{N}-{slug}` → `level-{N}` git tag in `project/myapp/`. Missing tag = SKIP, not failure.
- **Per-CodeFile classification**: REAL (real path) / PLACEHOLDER (`<timestamp>` / `<sha>`) / PSEUDO (filename has spaces or matches a known label like `Stack`, `Generator Command`) / GLOBAL (`~/.zshrc` etc., out of scope).
- **Comparison**: normalize whitespace, resolve placeholders to regex, every non-blank line of the level's snippet must appear (regex-matched) in the real file. Order-independent. Catches drift; tolerates legitimate simplification.
- **Out of scope (v1)**: `--fix` mode, per-step validation, CI gating. Run on demand via `bun run validate:codebase`.

### Implementation

- `scripts/validate-codebase.ts` (~270 lines, Bun script). Uses `Bun.Glob` for level discovery (cited <https://bun.com/docs/api/glob>), `Bun.$` for git shell calls (cited <https://bun.com/docs/runtime/shell>). Exports pure functions (`classifyFilename`, `tagForSlug`, `lineToRegex`, `normalize`, `compare`) so tests can import them.
- `scripts/__tests__/validate-codebase.test.ts` (~150 lines, 26 tests). Covers classification, slug-to-tag derivation, regex-meta escaping with placeholder substitution, normalization, line-containment match (incl. drift detection on single + multiple lines, whitespace tolerance, regex-meta safety).
- `src/lib/codebase-registry.ts`: added `getRegisteredLevelIds()` and `getLevelCode(id)` helpers so the validator can introspect the registry after triggering imports.
- `package.json`: new `validate:codebase` script.

### What the level claimed vs what Rails 8 actually does

L9 taught a token-based API auth pattern: `Session.token` column, `before_create { self.token = SecureRandom... }`, `Authorization: Bearer <token>` header parsing, JSON `{ token }` response. The level claimed `bin/rails generate authentication` produced this. **It does not.**

Source: <https://raw.githubusercontent.com/rails/rails/main/railties/lib/rails/generators/rails/authentication/authentication_generator.rb>. The Rails 8 generator produces:
- `User`, `Session`, `Current` models — Session with NO token column (just `belongs_to :user` + ip/user_agent strings).
- `app/controllers/concerns/authentication.rb` that uses `cookies.signed[:session_id]` for session lookup.
- `SessionsController#create` that calls `start_new_session_for(user)` (sets a signed cookie) and `redirect_to`.
- `helper_method :authenticated?` line — fails on `ActionController::API` (not present in API mode).

I confirmed the breakage at runtime: with the unmodified generator output, `POST /session` from our API-only myapp hits `NoMethodError: undefined method 'helper_method' for class ApplicationController`. **The vanilla Rails 8 auth generator is broken in API-only mode.**

### Decision

User confirmed: rewrite L9 to teach a real Rails 8 pattern, and ensure the generated app actually works (`rails s` should serve the API end to end). We picked: keep the generator as the starting point, then customize for API + bearer-token auth. This matches the level's existing pedagogical intent (token-based) while being honest about the customization required.

## Terminal auto-scroll regression

User flagged: auto scroll not working in L9's probe terminal. Root cause: `useEffect(() => { scrollRef.current.scrollTop = ...; }, [])` — the empty dependency array means the effect only runs on mount. New probe responses, animated output lines, etc. don't trigger the scroll. ProbeTerminal AND SimulatedTerminal both shipped with this bug. StressTestPanel was correct (deps `[resultCount]`).

### Fix

Both terminals now have `useEffect(() => { /* scroll */ }, [history, visibleLines])`. The deps aren't read inside the effect — they're scroll triggers. Biome's `useExhaustiveDependencies` rule wants to remove them, so each effect carries an explicit ignore comment with the intent. Without the trigger deps the effect goes back to mount-only, which is the bug.

## TDD enforcement hook

User requested an always-on TDD reminder. Added `.claude/hooks/enforce-tdd.sh` mirroring the existing `research-before-guessing.sh` pattern: a `UserPromptSubmit` hook that emits a JSON `additionalContext` payload prepending a non-negotiable TDD rule to every turn.

The injected text: "practice TDD. Before writing or modifying production code (functions, classes, methods, services, validators, helpers, scripts), write or update the test FIRST. Run the test, see it fail, then implement. ... Exceptions: docs, config files (.json, .yml, .toml), pure data/content definitions, snapshot updates that match real source, level component composition (rendering JSX without logic). For everything else, test first."

Reasoning over hard PreToolUse blocking: the codebase already has `.agents/rules/testing.md` ("write tests WHILE building, not after"), the validator + reveal-consistency tests catch level-shape regressions, and PreToolUse blocking on every Write/Edit would fight legitimate edits to non-logic files (level data, content, snapshot updates, JSX composition). ## Session 5: Curriculum-wide /api/v1 → /api un-versioning sweep

### Why

- **L1–L5**: no products / no HTTP. Untouched.
- **L6 (Routes)**: introduces `namespace :api do; resources :products` → `/api/products`. v1 dropped.
- **L7–L47**: use `/api/products` everywhere. Mechanical sweep.

### Mechanics

Three sed patterns covered ~98% of the work:

```

s|Api::V1::|Api::|g    # class namespaces (with trailing ::)
```
