# Session Log: 2026-05-03

## Codebase validator: design + build

After yesterday's Act 1 ground-truth pass shipped, evaluated three approaches for keeping level snapshots in sync with real myapp going forward:

1. **Per-level git-tag snapshots → JSON manifest → React consumer** (spike). Built end-to-end yesterday. Works, but 3.5MB manifest at scale, requires refactoring `getCodeFiles` reveal logic, and per-step granularity becomes its own problem.
2. **Continue manual ground-truth audits per act**. Brittle and high-touch.
3. **Validator script that compares hand-authored level snapshots against real myapp at git tags, reports drift**. Levels stay hand-authored and pedagogically tuned; correctness is enforced by a separate, runnable check.

User picked (3). Spike scaffolding from yesterday's experiment was reverted in commit a97dbcc to keep the slate clean.

### Design (docs/codebase-validator.md)

- **Scope (v1)**: validate only `registerLevelCode`'s output (the codebase viewer's snapshot). Per-step `getCodeFiles` content is a superset; deferred.
- **Mapping**: `act{X}-level{N}-{slug}` → `level-{N}` git tag in `project/myapp/`. Missing tag = SKIP, not failure.
- **Per-CodeFile classification**: REAL (real path) / PLACEHOLDER (`<timestamp>` / `<sha>`) / PSEUDO (filename has spaces or matches a known label like `Stack`, `Generator Command`) / GLOBAL (`~/.zshrc` etc., out of scope).
- **Comparison**: normalize whitespace, resolve placeholders to regex, every non-blank line of the level's snippet must appear (regex-matched) in the real file. Order-independent. Catches drift; tolerates legitimate simplification.
- **Out of scope (v1)**: `--fix` mode, per-step validation, CI gating. Run on demand via `bun run validate:codebase`.

### Implementation

- `scripts/validate-codebase.ts` (~270 lines, Bun script). Uses `Bun.Glob` for level discovery (cited <https://bun.com/docs/api/glob>), `Bun.$` for git shell calls (cited <https://bun.com/docs/runtime/shell>). Exports pure functions (`classifyFilename`, `tagForSlug`, `lineToRegex`, `normalize`, `compare`) so tests can import them.
- `scripts/__tests__/validate-codebase.test.ts` (~150 lines, 26 tests). Covers classification, slug-to-tag derivation, regex-meta escaping with placeholder substitution, normalization, line-containment match (incl. drift detection on single + multiple lines, whitespace tolerance, regex-meta safety).
- `src/lib/codebase-registry.ts`: added `getRegisteredLevelIds()` and `getLevelCode(id)` helpers so the validator can introspect the registry after triggering imports.
- `package.json`: new `validate:codebase` script.

### L8 observe phase: serializer shown as a "missing" node (also fixed)

User flagged a third L8 bug: the observe phase rendered a `Serializer` node with sublabel "Missing!" connected to the controller. Per `.agents/rules/pedagogy.md` ("show only what currently exists"), the observe phase must only show components that exist in the "before" state. The serializer doesn't exist yet — that's the whole premise of the level. Showing it as a placeholder is misleading.

Fix:
- Removed the `serializer` stage from `observeStages`.
- Removed the `controller -> serializer` edge from `OBSERVE_CONNECTIONS`.
- Removed the `serializer` entry from `STAGE_INSPECTOR_MAP` and `STAGE_DISCOVERY_MAP`.
- Removed `serializerSublabel` from `PROBE_PIPELINE_MAP` (the `'Bypassed!'` overlay was identical across all 3 probes anyway, so no per-probe differentiation lost — `responseBadge` already carries that).
- Dropped the now-orphan `no-serializer` discovery from `DISCOVERY_DEFS` (its only unlock source was the about-to-be-removed stage click). The 3 remaining discoveries stay 1:1 with the 3 probes.

The reward phase still includes the serializer (correctly — it exists by then). The probe responses already convey "no serializer exists" through raw, unfiltered JSON dumps — that's how the absence is communicated, per the pedagogy rule.

### L8 step-reveal off-by-one (separately reported by user, fixed together)

While reviewing, the user noticed that completing the last step of L8 (`update-controller`, step index 4) showed `product_serializer.rb` in the right panel instead of the controller. Root cause: `getCodeFiles` in `Level8Serializers.tsx` had guards `>= 1` through `>= 5`, but the right panel passes a 0-based completed-step index per the CLAUDE.md convention (max 4 in this 5-step level). The controller guard `>= 5` was unreachable during gameplay; only the codebase-viewer call (`registerLevelCode` passes `STEP_DEFS.length = 5`) ever satisfied it. There was also a duplicate `Gemfile` push at `>= 2`.

Fix: dropped the duplicate Gemfile, shifted the remaining guards down by 1 so each file appears at the correct step (Gemfile after step 1 = install-gem; BaseSerializer after step 2 = base-serializer; ProductSerializer after step 3 = define-attributes; controller after step 4 = update-controller). The validator continues to pass (registry call still matches all 4 real files).

### Regression tests for the L8 bug class

Added `scripts/__tests__/level-reveal-consistency.test.ts` (16 tests) to catch this class of bug going forward:

1. **Static: stranded `getCodeFiles` guards** — for every level component file, parses `STEP_DEFS` length, detects which step convention the render-side `getCodeFiles` call uses (`stepper.currentStep` = 0-based, max `length - 1`; `stepper.furthestStep` = 1-based, max `length`), finds the largest `>= N` guard inside the file, and asserts the guard is reachable in-game.
2. **Runtime: duplicate filenames in `registerLevelCode` output** — for every registered level, asserts no `CodeFile.filename` repeats.
3. **Static: observe phase shows only what currently exists** — for every level, scans for textual fingerprints of the "render a placeholder for a thing the build phase will introduce" anti-pattern: inspector titles ending `(Missing)` / `(Missing!)`, stage labels ending `(Missing)` / `(Missing!)`, bare `sublabel: 'Missing'` / `'Missing!'`. Distinguishes legitimate "No X" sublabels (which describe an absent property of an existing node) from the bug.

All three tests use a **baseline** (`KNOWN_STRANDED_GUARDS`, `KNOWN_DUPLICATE_FILENAMES`, `KNOWN_OBSERVE_PHASE_PLACEHOLDERS`) of levels that already exhibit the bug, so the test catches *new* regressions while documenting existing technical debt to fix later. Each test also fails if a baseline entry has been fixed but not removed (no stale entries).
