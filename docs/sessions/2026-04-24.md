# Session: 2026-04-24

## Summary

Completed the e-commerce schema canonicalization sweep across all levels that
still referenced the old blog-post schema (`title`/`body`/`published_at`,
`Post`/`PostList`/`PostsController`). The reframe to e-commerce had been
applied unevenly: some content files (L11, L13, L7) had been migrated,
but the foundational L3 schema, the controllers/serializers downstream, and
all of Act 4's performance levels still used the old vocabulary.

## Decisions

**Canonical Product schema (locked in):**
- `name:string` (was `title`)
- `description:text` (was `body`)
- `price:decimal` (new; replaces `published_at` in the L3 generator slot)
- `listed_at:datetime` (was `published_at`; appears starting at L11/L19)

`listed_at` is NOT in L3's three-slot drag-and-drop. It enters the model
later when callbacks (L11) and query filters (L19) need a "is this product
currently for sale" flag. This matches L11 content's existing
`before_save :set_listed_at, if: :listing?` pattern.

**Variable / class naming:**
- Variable `posts` → `products`
- Service `PostList` → `ProductList`
- Controller `PostsController` → `ProductsController`
- Filename `post_list.rb` → `product_list.rb`
- Path `/api/posts` → `/api/products` (Act 4 narrative paths)
- `/jobs/export_posts` → `/jobs/export_products`
- Migration class `AddReviewsCountToPosts` → `AddReviewsCountToProducts`
- Migration filename `add_reviews_count_to_posts.rb` → `add_reviews_count_to_products.rb`

**Query Object filter naming (L19):**
- `params[:published]` → `params[:listed]`
- `params[:author_id]` → `params[:seller_id]`
- Method `.published(flag)` → `.listed(flag)` to match `listed_at` field

## Notes

- **L4 was already correct.** A prior audit had flagged `product.name` in
  Level 4's CRUD step as a typo. It was not a typo: L4 had been migrated to
  the e-commerce schema. L3 was the file out of sync. The fix flowed in the
  opposite direction (update L3 to match L4), not the direction the audit
  suggested.
- **`listed_at` deferred to later levels.** Adding it to L3 would have
  required reshaping the drag-and-drop UI (which has exactly three slots
  matching three available types). L3 stays at three fields; `listed_at`
  enters the model in L11 callbacks where it has a clear narrative reason
  to appear (the "list a product for sale" callback).
- **L24 had pre-existing `key={i}` warnings** unrelated to the schema
  sweep. Per CLAUDE.md "you touched the file, you own every error in it"
  these were fixed in the same pass with composite stable keys.
- The narrative drift was worse than initially scoped — the audit caught
  the L3 model/table mismatch but missed that L19's `posts` variable,
  L23's PostList service, and the entire Act 4 test suite were also stuck
  on the old vocabulary. The full sweep ended up touching 13 files.
