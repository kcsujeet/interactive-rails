## Batch 30: L33 Transactions depth (race conditions, isolation, deadlocks, side effects, savepoints)

### Audit findings

L33 currently teaches:

- Wrap related writes in `ActiveRecord::Base.transaction do ... end`
- Use `raise ActiveRecord::Rollback` for business-rule aborts
- Service-object pattern with `Result = Data.define(...)`

What it does NOT teach (each one a real production-trip bug):

1. **Race condition inside the transaction.** The example body has
   `raise ActiveRecord::Rollback if user.credits < @cost` followed by
   `user.credits -= @cost; user.save!`. Two concurrent requests can
   both pass the check, both decrement, and the user's credits go
   negative. The transaction guarantees atomicity, not isolation from
   concurrent writers. The fix: pessimistic `lock!` (SELECT FOR UPDATE)
   or atomic conditional UPDATE.
2. **Isolation levels.** PostgreSQL defaults to READ COMMITTED. For
   multi-row consistency, code should explicitly request
   `isolation: :repeatable_read` or `:serializable`. Not mentioned.
3. **Deadlock retry.** Two transactions touching rows in different
   orders trigger `ActiveRecord::Deadlocked`. Production code retries
   2-3 times with exponential backoff. Not mentioned.
4. **No I/O inside transactions.** Holding row locks and a connection
   from the pool during HTTP calls or email sends is one of the most
   common pool-exhaustion bugs at scale. Not mentioned.
5. **`after_commit` vs `after_save` for side effects.** Enqueueing a
   job from `after_save` lets the worker dequeue it before the COMMIT
   and hit `RecordNotFound`. Always use `after_commit`, or Rails 7.2+'s
   `enqueue_after_transaction_commit`. Not mentioned.
6. **Nested transactions and savepoints.** Rails treats a nested
   `transaction do` block as a no-op by default. The inner `Rollback`
   is silently swallowed and the outer transaction commits anyway. The
   `commonMistakes` list flagged this without explaining the fix
   (`requires_new: true`). Not mentioned in the explanation.
7. **Lock-free atomic UPDATE for counter operations.** For pure
   counter/balance updates, the conditional `update_all` pattern is
   the right tool: no row lock, no transaction needed, the database
   enforces the invariant. Not mentioned.

### Decision

Append seven new sub-sections to `learningContent.conceptExplanation`,
following the structure used in Batch 22 (L40) and Batch 23 (L51).
Build steps and stress scenarios untouched (this is a content depth
addition, not a level redesign). The mechanic still teaches the
core concept; the briefing screen now teaches the depth that makes
the level production-relevant.

The seven additions, in order:

1. **Race conditions (the check-then-set bug).** Names the bug
   explicitly, shows both fixes (pessimistic `lock!` and lock-free
   atomic UPDATE).
2. **Isolation levels.** Names the Postgres default (READ COMMITTED),
   shows the syntax for bumping to `:repeatable_read` and
   `:serializable`, flags `SerializationFailure` as a retry trigger.
3. **Deadlocks (retry with backoff).** Working snippet with
   exponential backoff bounded at 3 retries.
4. **Never do I/O inside a transaction.** States the rule, names
   connection-pool exhaustion as the cascading failure mode.
5. **`after_commit` vs `after_save`.** WRONG / RIGHT example, plus
   the Rails 7.2+ `enqueue_after_transaction_commit` alternative.
6. **Nested transactions and savepoints.** Names the silent-swallow
   default, shows `requires_new: true` for partial rollback.

Six new entries in `commonMistakes`, each describing a specific
production-trip bug. The existing entry on nested transactions
expanded to name `requires_new: true` so the mistake list is
self-contained.

Four new `furtherReading` links: Rails Pessimistic Locking docs,
PostgreSQL Transaction Isolation docs, Rails 7.2
`enqueue_after_transaction_commit` API. The existing Active Record
Transactions link stays.

## Batch 31: L41 Middleware & Rack depth (CurrentAttributes, traceparent, Rack::Attack, lograge, /up exemption, Rack 3 headers)

### Audit findings

L41 currently teaches custom Rack middleware via three example
classes: `RequestIdTracker`, `RequestLogger`, `BotDetector`. The
mechanic is correct; the example code has billion-dollar SaaS
gaps a staff Rails engineer would call out:

1. **`Thread.current` for per-request state.** Works on Puma. Breaks
   on fiber-based servers like Falcon, where many fibers share one
   thread and stomp each other's thread-locals. Rails ships
   `ActiveSupport::CurrentAttributes` for exactly this; the manual
   `ensure Thread.current[:x] = nil` cleanup goes away because Rails
   resets `Current` between requests automatically.
2. **`X-Request-Id` only.** Fine for one app, insufficient once
   requests cross services. Modern systems use the W3C
   `traceparent` header (and `tracestate`); OpenTelemetry, Datadog,
   Honeycomb, and Sentry all read it directly.
3. **`Rails.logger.info({ ... }.to_json)` for structured logging.**
   Fragile (one newline in a value breaks the parser) and bypasses
   Rails' tagged-logging infrastructure. Production answer: `lograge`
   for one structured line per request, or `Rails.logger.tagged` for
   per-request prefixes.
4. **Hand-rolled `BOT_PATTERNS` regex.** Maintenance trap: every new
   bot is a regex edit. `Rack::Attack` is the production standard,
   supports throttling, banning, and built-in safe-lists for known
   good crawlers. Plus: rate-limit responses should be `429 Too Many
   Requests` with a `Retry-After` header, not `403`.
5. **No health-check exemption.** Rails 8 ships `/up`. The load
   balancer hits it every few seconds; if logging middleware logs
   `/up` and bot detection 403s curl-style user agents, the LB
   floods the logs and eventually trips the bot detector.
6. **Rack 3 header casing.** Rack 3 requires response header keys
   to be lowercase strings. The example writes
   `headers['X-Request-Id']`; under Rack 3 that should be
   `headers['x-request-id']`. Works today via deprecation shim,
   will break in future Rack versions.
7. **`insert_before 0` caveat.** Puts a middleware above
   `HostAuthorization`, which means it runs on requests that would
   otherwise be rejected for a bad Host header. Fine for request-id
   assignment, wrong for anything that assumes the request has been
   validated.

### Decision

Append seven sub-sections to `learningContent.conceptExplanation`,
following the same structure as Batch 30 (L33). The build steps
and example code (`Thread.current`, `BOT_PATTERNS`, etc.) stay as
they are. The level still teaches the mechanic correctly; the
briefing screen now teaches what production-safe equivalents look
like and when to graduate from the starter pattern.

Seven new entries in `commonMistakes`, each describing a specific
production-trip bug (Thread.current on Falcon, logging /up, hand-
rolled regex bot detection, 403 instead of 429, mixed-case Rack 3
headers, fresh request id per hop, bare `.to_json` logging).

Four new `furtherReading` links: `ActiveSupport::CurrentAttributes`
API docs, `Rack::Attack` GitHub, the W3C Trace Context spec, and
the `lograge` gem.

### Why this is content-only

The L41 build steps still teach the high-level concept (write a
class with `initialize(app)` + `call(env)`, insert it into the

## Batch 32: L36 Encrypted Attributes depth (frequency analysis, migration procedure, key rotation, KMS, what encryption does NOT do)

### Audit findings

L36 covers the basics of `encrypts` (deterministic vs non-deterministic,
keys in credentials, transparent reads/writes) but has staff-engineer
gaps:

1. **Frequency analysis is named but not explained.** The level says
   deterministic encryption is "vulnerable to frequency analysis"
   without explaining what that means. A senior reviewer would expect
   the example: low-cardinality fields like `status` leak the value
   distribution even when encrypted.
2. **Migration procedure is glossed over.** The brief mention of
   `User.find_each { |u| u.save!(validate: false) }` is wrong: that
   does not force re-encryption unless the attribute is dirty. The
   real procedure uses `support_unencrypted_data: true` during the
   backfill window, batches the writes, then removes the flag.
3. **Key rotation is one line.** Production rotation is a multi-step
   procedure: deploy with `previous` keys configured, re-encrypt,
   verify, remove `previous`. SOC 2 / HIPAA / PCI auditors ask about
   cadence; the level should signal that.
4. **No HSM / KMS pattern.** Strict compliance shops cannot keep keys
   in Rails credentials. The fetch-from-KMS-at-boot pattern is the
   production answer.
5. **What encryption does NOT replace** is missing entirely: row
   counts still visible, query patterns still leak (`WHERE email = ?`
   tells the attacker you queried by email), full-text search becomes
   impossible. These are the surprises that bite in production.

### Decision

Append five sub-sections to `learningContent.conceptExplanation`:
frequency analysis, migration procedure (with the
`support_unencrypted_data` flag), key rotation procedure, external
key management (KMS), and "what encryption does NOT replace."

Six new entries in `commonMistakes`, each describing a specific
production-trip bug (no `support_unencrypted_data` during backfill,
`update_all` skipping the encryptor, deterministic on low-cardinality
fields, `LIKE '%foo%'` after encryption, key in committed credentials,
no rotation cadence).

Two new `furtherReading` links: AWS KMS and GCP KMS.

## Batch 33: L13 Testing depth (WebMock/VCR, time helpers, async job assertions, parallel testing, fixtures-vs-factories tradeoff, system specs, coverage policy, test smells)

### Audit findings

L13 currently covers RSpec, FactoryBot, and request specs. Solid
foundation. Staff-engineer gaps:

1. **No external API stubbing.** The moment a service calls Stripe
   or SendGrid, the test suite either calls real APIs (slow, flaky,
   leaks secrets) or breaks. WebMock + VCR is the production answer.
2. **No time helpers.** `Time.now + 1.day` in tests is the source of
   "passes Tuesday, fails Friday at midnight" flakes. Rails ships
   `travel_to`, `freeze_time`, `travel`. Critical for tokens, expiry,
   scheduled jobs.
3. **No background job test patterns.** Active Job test adapter +
   `have_enqueued_job` + `perform_enqueued_jobs` is how you assert
   async behavior. The level does not mention the test adapter or
   the matchers.
4. **No parallel testing.** Past ~500 tests, suite time becomes the
   feedback bottleneck. Rails has built-in `parallelize` but L13
   does not surface it.
5. **No fixtures-vs-factories tradeoff.** The level recommends
   factories absolutely. At 5K+ tests, all-factories suites become
   IO-bound; major Rails shops (Shopify, GitHub) mix fixtures for
   "every test needs one of these" entities and factories for
   specific scenarios.
6. **No system specs.** Even API-only apps have end-to-end flows
   (OAuth callbacks, webhook deliveries) that need real-browser
   tests.
7. **No coverage policy.** SimpleCov as a CI gate at 90% encourages
   tests-for-coverage; the right policy is "untested files at 0%
   are visible" plus per-file gates on new files only.
8. **No test smells list.** `sleep N`, mocking your own classes,
   non-deterministic Faker, order-dependent state. Each is the
   source of a recurring class of CI flakes.

### Decision

Append eight sub-sections to `conceptExplanation`: WebMock/VCR,
time/randomness helpers, background job patterns, parallel testing,
fixtures-vs-factories tradeoff, system specs, coverage as a
guardrail, test smells to watch for.

Eight new entries in `commonMistakes` mirror the depth additions.
Four new `furtherReading` links: WebMock, VCR, Rails parallel
testing guide, ActiveSupport time helpers.

## Batch 34: L19 Query Objects depth (Pundit composition, eager-load embedding, cursor pagination, EXPLAIN, subquery shape, OR composition, read replicas)

### Audit findings

L19 already covers the basic pattern (chainable POROs returning
`ActiveRecord::Relation`). Staff-engineer depth gaps for billion-
dollar SaaS:

1. **No mention of Pundit composition.** A query object that uses
   `Product.all` as base scope silently bypasses authorization. The
   right pattern is `ProductQuery.new(policy_scope(Product))`.
2. **N+1 left to the caller.** The query object returns 100 products;
   if the controller does `.map(&:reviews)`, N+1 happens. Embed
   `.includes` as a chainable filter so the dependency is explicit.
3. **No cursor pagination.** `OFFSET 50_000` becomes the slowest
   endpoint in the app at scale. Cursor pagination via indexed
   `WHERE id > ?` is O(1) regardless of depth.
4. **No EXPLAIN as a habit.** "Fast in dev because the table has 200
   rows; sequential-scans 50M in prod" is the most common scaling
   bug. `Product.all.explain` before merging a new query method
   catches it.
5. **No subquery composition note.** The classic mistake of
   `User.where(...).pluck(:id)` then `.where(seller_id: ids)` is a
   round trip through Ruby memory. `where(seller_id: User.where(...))`
   pushes the join down to Postgres.
6. **OR composition is missing.** `where` is implicitly AND;
   composing OR requires `.or` or `.merge` with caveats around
   `includes` being silently dropped.
7. **No read replica routing.** At billion-dollar SaaS scale, query
   objects should know how to route SELECTs to a replica via
   `connected_to(role: :reading)`. Plus the replication-lag gotcha
   for write-then-read flows.

### Decision

Append seven sub-sections to `conceptExplanation`. Six new
`commonMistakes` entries. Three new `furtherReading` links: pagy,
Active Record multiple databases guide, PostgreSQL EXPLAIN docs.

## Batch 35: L20 Error Handling depth (StandardError-not-Exception, Sentry/Honeybadger, errors.details, more status codes, RFC 9457 Problem Details, service-Result vs rescue_from boundary)

### Audit findings

L20 covers `rescue_from` with the basic shape `{ error: { code,
message, details } }`. Staff-engineer gaps:

1. **No "rescue StandardError, never Exception" rule.** Catching
   `Exception` swallows `Interrupt` (Ctrl+C), `SystemExit`, and
   `NoMemoryError`. Common interview question, common production
   bug.
2. **No Sentry / Honeybadger / Bugsnag integration.** Logging
   `exception.message` plus 20 backtrace lines is the bare minimum.
   At scale you want full backtrace, request context, user id,
   request id, and grouping across similar errors. The
   `internal_server_error` handler should call
   `Sentry.capture_exception(...)` with extras.
3. **No `request_id` in the response.** When a 500 happens, the
   client should be able to quote the request id to support, and
   support should be able to find the line in the log. Currently
   the response shape doesn't include it.
4. **`errors.messages` vs `errors.details`.** The current example
   uses `errors.messages` (English strings). Modern API clients
   want `errors.details` (structured codes) for i18n.
5. **Limited status code coverage.** The example handles 404 / 422
   / 400 / 403 / 409 / 500 but not 429 (rate limited), 503 (circuit
   breaker open), 408 (request timeout), 410 (gone, for sunset
   endpoints). Each is a distinct signal to the client.
6. **No RFC 9457 Problem Details mention.** The IETF standard for
   error responses is `application/problem+json` per RFC 9457.
   Greenfield public APIs should ship Problem Details from day one.
7. **Service-Result vs `rescue_from` boundary unaddressed.** The
   Act 3 lineage uses `Result` objects for explicit failure paths;
   mixing that with `rescue_from` requires a documented convention
   so error responses are consistent across endpoints.

### Decision

Append six sub-sections to `conceptExplanation`: StandardError-not-
Exception, exception reporting + request context, validation error
shape via `errors.details`, additional HTTP status codes, RFC 9457
Problem Details, service-Result vs `rescue_from` boundary.

Seven new entries in `commonMistakes`. Three new `furtherReading`
links: RFC 9457, Sentry for Rails, errors.details API docs.

## Batch 36: Foundation acts (L1-L8) sweep against rails-conventions.md

### Audit findings

Read all 8 foundation levels and cross-checked against
`.agents/rules/rails-conventions.md`. L1, L2, L3, L5, L7 are clean
(setup or pure-foundation; no convention violations big enough to act
on). Three files have real production-safe-default gaps:

**L4 (CRUD):**
1. No bang-vs-non-bang explanation. `Product.create` returns the
   record whether or not it actually saved; `Product.create!` raises.
   This bites anyone running scripts/seeds and getting silently-invalid
   records that propagate downstream.
2. Example uses `price: 999.99` (Ruby Float literal) for a decimal
   column. Floats lose precision; production code passes
   `BigDecimal('999.99')` or the string `'999.99'`.
3. `update_column` is mentioned in a comment but never warned about.
   It skips validations, callbacks, AND the `updated_at` timestamp,
   which is almost always wrong.

**L6 (Controller):**
1. `Product.find(params[:id])` is repeated three times in the example
   (`show`, `update`, `destroy`). The conventional Rails idiom is
   `before_action :set_product, only: [...]` so the lookup lives in
   one place. Worth introducing at L6 since the player will reuse the
   pattern in every subsequent controller.
2. The example uses `params.require(:product).permit(...)` and the
   comment says "Rails 8 introduces params.expect()." Per
   `rails-conventions.md`, `expect` IS the production default. The
   level can keep the `require/permit` example for compatibility with
   older controllers but should explicitly flag that all new code
   skips straight to `expect` (which L14 teaches).

**L8 (Associations):**
1. No `inverse_of:` on bidirectional associations. Rails 5+
   auto-detects in common cases but fails when the FK column doesn't
   match (`reviewer_id` instead of `product_id`) or when the
   association uses a custom `class_name:`. The failure mode is
   in-memory parent and child drifting out of sync, which is hard to
   debug.
2. `dependent:` choices not enumerated. The example shows
   `dependent: :destroy` without naming `:delete_all`, `:nullify`,
   or `:restrict_with_error`. Each is a real decision and a bare
   `has_many` with no `dependent:` orphans children silently.
3. FK constraint mentioned in passing (`foreign_key: true`) but the
   defense-in-depth argument (model `dependent:` + database FK
   constraint) is not made. Without the FK constraint, a stray
   `Product.delete` (skipping callbacks) leaves orphans the database
   has no idea about.

### Decision

Targeted depth additions to L4, L6, L8 only. L1, L2, L3, L5, L7
unchanged.

## Batch 37: L39 / L38 / L34 staff-engineer depth (webhooks, external APIs, locking)

A third pass of high-stakes uncovered topics. Each level had
production-trip gaps that a senior reviewer would call out.

### L39 Webhooks & Idempotency

Already covers signature verification, idempotency-via-events-table,
async processing. Missing six production-critical concerns:

- **Raw body, not parsed body.** Stripe signs the exact bytes; Rails
  parses JSON before \`params\` is populated, which can re-order keys
  and break the HMAC. Fix: use \`request.raw_post\`.
- **Replay protection.** \`construct_event\` accepts a \`tolerance\`
  argument; without it, a captured signed payload verifies forever.
- **Constant-time comparison.** Hand-rolled HMAC verification with
  \`==\` leaks timing information that an attacker can exploit byte
  by byte. Use \`ActiveSupport::SecurityUtils.secure_compare\`.
- **Trust signed payloads as a notification, not as truth.** For
  money-moving events, re-fetch the resource from Stripe before
  processing. Webhook payload may be stale (after retries) or
  planted by an attacker who breached the DB.
- **Out-of-order events.** Stripe makes no ordering guarantee.
  \`Payment.find_by!(...)\` will fail when the create event arrives
  after the succeeded event. Use \`find_or_create_by!\` or a state
  machine that tolerates out-of-order arrival.
- **Per-endpoint secrets.** Each Stripe endpoint has its own signing
  secret; one global \`ENV\` var does not scale.
- **Retention policy.** Without cleanup, the webhook_events table
  grows forever and lookups slow down.

Six new sub-sections, six new \`commonMistakes\` entries, two new
\`furtherReading\` links (secure_compare API, raw_post API).

### L38 External APIs

Already covers timeouts, retries, circuit breaker. Missing seven
production concerns:

- **Idempotency-Key explained, not just used.** The example sets it
  but does not explain that the key must be stable across retries
  (derived from the order id, not a fresh UUID per attempt). With
  the key, the "don't retry POST" rule becomes "you CAN retry POST
  with an Idempotency-Key."
- **Connection pooling.** New TLS connection per call costs 100-300ms
  of handshake. \`net-http-persistent\` keeps connections warm.
  Pool exhaustion under thread contention is its own bug class.
- **Bulkhead pattern.** Per-service connection pool prevents one
  slow dependency from consuming all threads before the breaker
  trips.
- **Logging vs PII.** Default Faraday logger is too permissive for
  production. Filter sensitive headers, never log full bodies.
- **Credentials over ENV.** Encrypted credentials file rotates better
  than ENV vars (which require a deploy and leak forever once leaked).
- **TLS verification.** \`VERIFY_NONE\` shipped from dev to prod is a
  hard-to-detect security regression. Fix the cert chain, do not
  disable verification.
- **Failure budget / SLO awareness.** Breaker thresholds without
  baseline measurement trip against healthy services.

Seven new sub-sections, seven new \`commonMistakes\` entries, three
new \`furtherReading\` links (Stripe Idempotency, net-http-persistent,
Bulkhead pattern).

### L34 Locking

Already covers optimistic vs pessimistic. Missing eight concerns:

- **Lock-free atomic UPDATE.** For pure counters, neither lock is
  the right tool; the conditional UPDATE wins (cross-references
  L33's pattern).
- **Lock ordering.** Two transactions that lock multiple rows in
  different orders deadlock. Pick a canonical order codebase-wide.
- **\`lock_timeout\` setting.** Without it, FOR UPDATE waits forever
  on a stuck transaction.
- **\`SKIP LOCKED\`** for queue-style "next available row"
  processing; without it, every worker fights every other.
- **\`NOWAIT\`** for fail-fast when the row is already locked.
- **Advisory locks** for non-row resources (cron jobs that should
  not overlap, distributed leader election).
- **Optimistic lock retry pattern.** Rescue + reload + reapply +
  save again, capped at a small number of retries. Missing from
  the example.
- **CHECK constraints as the safety net.** \`stock_count >= 0\` at
  the database layer catches bugs that bypass application logic.

Eight new sub-sections, seven new \`commonMistakes\` entries, four
new \`furtherReading\` links (PG row locks docs, PG advisory locks
docs, with_advisory_lock gem, PG CHECK constraints).

### L42 Kamal deployment

Already covers `kamal init`/`setup`/`deploy`/`rollback`, `deploy.yml`
shape, secrets via `.kamal/secrets`. Missing eight production
concerns:

1. **Database migrations during deploy.** The pre-deploy hook runs
   migrations against production before the traffic shifts. Backward-
   compatible migrations (the L45 rules) are the prerequisite.
2. **Accessory containers.** Solid Queue worker, Solid Cable, and the
   recurring scheduler are not in the web service; they are
   accessories that share the same image. Without them, `perform_later`
   queues forever.
3. **Asset precompilation.** Propshaft compiles fingerprinted assets
   at image build time, served by Kamal's proxy with long cache
   headers.
4. **Image tag immutability.** The git SHA is the contract. Re-tagging
   breaks rollback semantics.
5. **Logs and debugging.** `kamal app logs`, `kamal app exec`,
   `kamal proxy logs`. Surface them, not raw `docker exec`.
6. **Multi-environment setup.** `-d staging` / `-d production` switches
   the deploy file. Same SHA, different secrets, different servers.
7. **Smoke tests after deploy.** Healthcheck `/up` proves boot;
   pre-traffic hooks prove dependent services reachable.
8. **Master key handling.** `RAILS_MASTER_KEY` is the trust boundary
   on the encrypted credentials file. Never commits, never logs.

Eight new sub-sections appended to `conceptExplanation`. Seven new
`commonMistakes` entries. Three new `furtherReading` links (Kamal
accessories, hooks, proxy/SSL).

Plus pre-existing em-dashes in the file fixed (lines 60, 61).

### L37 Real-Time

Already covers Solid Cable, channels, broadcast pattern. Missing six
concerns:

1. **Token authentication for SPA / mobile clients.** Cookies do not
   always work cross-origin; bearer-style tokens via signed query
   param are the production answer.
2. **Broadcast IDs, not full payloads.** Tiny event with id; client
   fetches detail through the cached HTTP path. Avoids audit-log
   bloat and aligns WebSocket and HTTP semantics.
3. **Per-user subscribe rate limit.** Each subscription holds a
   server resource; without a limit, a malicious client can open
   thousands.
4. **Solid Cable retention tradeoff.** Long retention helps
   reconnection replay but bloats the cleanup table; short retention
   saves cost but drops late-joining clients.
5. **When to graduate from Solid Cable.** AnyCable for hundreds of
   thousands of concurrent connections. Decision is volume-driven;
   do not graduate prematurely.
6. **Reconnection ordering.** WebSockets do not guarantee in-order
   delivery across reconnects. Include a sequence number for
   ordering-sensitive flows.

Six new sub-sections, five new `commonMistakes` entries, one new
`furtherReading` link (AnyCable).
