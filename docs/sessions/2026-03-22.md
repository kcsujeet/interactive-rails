# Session: 2026-03-22

## Level 38 External APIs: Complete Redesign

### Design Skill Updates (3 new additions)

1. **Principle 13: Nodes at the same visual level must be the same kind of thing.** If all other nodes are systems (App, Database, Stripe), middleware concepts (timeout, retry, circuit breaker) don't belong as peer nodes. They should appear as sub-elements inside the parent node they run on. Case study: L38's original design had 5 peer nodes (App -> Timeout -> Retry -> Circuit Breaker -> Stripe), implying middleware was separate infrastructure.

2. **Design Probes and Reward Scenarios Together as Pairs (process rule).** The most common design mistake is designing observe probes in Step 3 and reward scenarios in Step 5 as separate exercises. This causes reward scenarios to become feature demos instead of story continuations. Fix: design each probe alongside its matching reward scenario before moving to the next probe.

3. **visualization-examples.md**: Added checklist items for both new principles.

### Level 38 Component Rewrite

**Removed:**
- Activate phase (was: observe -> build -> activate -> reward)
- FlowConnector-based horizontal gate pipeline
- Gate state that was identical for all 3 probes

**Added: React Flow visualization with 2 custom nodes**
- `AppServerNode` (large, left): Thread pool with 5 colored bars (green=available, red=blocked), status text, queue counter. In reward phase: expands to show middleware sub-panels (Timeout, Retry, Circuit Breaker) inside the node.
- `StripeNode` (compact, right): Status badge, response indicator.
- `ApiEdge`: Animated dots with reversible direction, dynamic labels.

**Observe probes (story-driven, with info modals):**
1. POST create payment (slow response) -> discovers no-timeout, thread-blocking. Thread pool drains as request hangs.
2. GET check payment status (Stripe 503) -> discovers no-retry. Changed from POST to GET so retry middleware can fix it in reward.
3. Black Friday traffic (Stripe outage) -> discovers cascade-failure. All 5 threads blocked, GET /products can't proceed.

**Reward scenarios (designed as pairs with probes):**
1. POST create payment (with timeout) - same flow, timeout kills at 10s
2. GET check payment status (with retry) - same flow, retry catches 503
3. Black Friday traffic (with circuit breaker) - same flow, circuit opens after 5 failures
4. POST charge (fast response) - baseline happy path (reward-only)
5. POST charge (400 bad params) - error filtering (reward-only)

**Build phase fixes:**
- Code preview uses `isCurrentStepCompleted ? currentStep : currentStep - 1` (no answer leaks)
- ErrorFeedback positioned above options
- StepDef uses `title` not `label`
- OptionCard uses correct props (`name`, `mono`, `selected`)
- StepProgress uses `stepper.steps`
- DiscoveryChecklist uses `discoveryGating.discoveries`

### Key Learnings

- When two nodes look the same but represent fundamentally different categories (systems vs code), the player assumes they're peers. Show code-level concepts (middleware, policies) as sub-elements inside the node they run on.
- Designing probes and reward scenarios separately causes unconscious drift from "replaying the story" to "demoing features." Design them as pairs.
- Changed probe 2 from POST to GET because POST can't be retried (non-idempotent), which means the retry fix wouldn't apply to the same scenario.
