## Session 3: Redesign L16 Observe Phase with Request Simulation Timeline

User feedback on ScenarioCards + abstract blocks: "i still don't like it. I don't know what I'm doing here... the animation finishes and what is the animation trying to convey to me? think deeper. focus on level 16 for now."

### Problem with previous approach
Abstract boxes labeled "User Creation", "Welcome Logging" etc. don't teach anything. Clicking a scenario card triggers an animation through these boxes, but the animation has no visible consequence. The player can't see WHY the fat controller is a problem.

### New approach: Request Simulation Timeline

**File:** `frontend/src/features/act3-clean-architecture/components/Level16ServiceObjects.tsx`

Complete rewrite of the observe phase. The center panel now shows a vertical timeline of 4 sequential controller steps, each with actual Ruby code and timing:

1. **User Creation** (200ms): `User.create!(name: params[:name], email: params[:email])`
2. **Welcome Logging** (800ms): `WelcomeLogger.new.log(user)` with HTTP call to logging service
3. **Default Preferences** (1200ms): `UserPreference.create!(user:, theme: 'light', ...)`
4. **Token Generation** (1000ms): `SecureRandom.hex(32)` + `user.update!(api_token: token)`

Each step is clickable for inspection (fires discoveries). Two simulations:
- **Normal request**: All steps succeed sequentially, accumulating to 3200ms. Result: "User waits 3.2 seconds. All side effects run sequentially."
- **Logger crashes**: Step 1 succeeds (green), step 2 fails (red, "Connection refused!"), steps 3-4 go dark ("Never reached"). Result: "User created but has no preferences and no session token. Broken state."

Key design decisions:
- Actual code snippets, not abstract labels
- Accumulated timer showing real-time cost
- Simulation results persist with clear banner explaining the consequence
- Player sees the problem through consequences, not just structure

Build passes cleanly.
