## Session 4: L41 deep pass (CORS mechanism-honest redesign)

### Why (the last P0 from the teaching audit)

The old L41 taught CORS backwards: a "CORS Middleware" gate node sat BETWEEN the browser and the API, requests visually stopped at it, the API showed "not reached", and the observe frames said "Response blocked before reaching app". Its content said "the browser blocks it (the request never reaches your code)". Per MDN, that is the opposite of the mechanism: simple requests reach Rails, Rails does all the work, and the BROWSER discards the response on the way back. The gate node also violated show-only-what-exists (the middleware was rendered in observe as a "(not installed)" placeholder).

### Doc verification (Step 0.5, fetched 2026-07-10)

- MDN CORS guide: simple requests are sent to the server; the browser makes the response unavailable to the invoking script when Access-Control-Allow-Origin is missing. Preflighted requests (DELETE) send OPTIONS first "to determine if the actual request is safe to send"; a failed preflight means the actual request is never sent. CORS protects the browser user; non-browser clients ignore it.
- rack-cors source (cyu/rack-cors lib/rack/cors.rb `call`): preflights are answered by the middleware directly (`return [200, headers, []]`, the app is never called); actual requests ALWAYS pass through (`@app.call env`), allowed origin or not; disallowed origins simply get no headers added. So even in the reward phase, an evil-origin GET runs on the server and the visitor's browser is what withholds the response.
- rack-cors latest version confirmed 3.0.0 (rubygems API); terminal output updated from the invented 2.0.2.
- Before-state ground truth (project/myapp, rails new --api): config/initializers/cors.rb ships fully commented out and Gemfile carries `# gem "rack-cors"`. The observe preview now shows the real stub (sample block elided so the DSL never leaks in observe).

### Design decisions (design-level worksheet outputs)

- **Narrative (Step 1).** Problem in one sentence: the storefront moved to its own origin and every page went blank, while the API returns nothing but 200s, because the browser discards responses that carry no permission header. Actor table: Customer (storefront), Admin (dashboard delete), Developer (curl), Visitor-on-evil-page (reward extras), Browser (the enforcer), Rails API (healthy throughout). Topology: Browser <-> Rails API with an explicit request leg and response leg; no gate node, because none exists on the wire.
- **Smell test (written):** the player fires "Load the storefront from the new origin" and watches the request reach Rails, Rails return "200 OK, 42 products", and the browser stamp DISCARDED on the way in, with "Storefront rendered 0 products" in the console strip. Reaction: "the server did all that work and my customers see an empty shop; I have to grant the permission." PASS: the damage is customer-concrete and the villain (the missing response header) is visible at the exact place it acts.
- **Probe playthrough (written, per frame):** (1) fetch-products: browser amber GET -> request dots -> API green "ran the query, returned 42 products" 200 OK -> red response dots "no Access-Control-Allow-Origin" -> browser red DISCARDED, console "Storefront rendered 0 products". (2) preflight-delete: browser amber OPTIONS "DELETE needs permission first" -> API red 404 "no route answers OPTIONS" -> reply with no permission -> browser red NEVER SENT, "delete button does nothing". (3) curl-bypass: client card swaps to terminal chrome -> API 200 -> full JSON delivered, "no browser, no CORS check". Three distinct shapes (work-discarded, never-sent, unaffected), test-enforced distinct.
- **Reward mechanism-honesty:** rack-cors renders as a strip INSIDE the Rails API card (it is Rack middleware, not a network hop; the old peer-node layout violated nodes-same-kind-of-thing). The three probes replay as paired scenarios ("Same ..." stories); two evil-origin extras show what stays shut, honestly: evil-read still returns 200 from the app and the VISITOR'S BROWSER withholds it (badge WITHHELD, never "not reached"); evil-delete dies at the answered preflight, "the DELETE never leaves the page". The preflight-delete reward plays the full two-round-trip flow (OPTIONS answered by the middleware without touching the router, then the real DELETE). Left-panel legend: "Blocked: the visitor's browser withholds it (the server still ran)".

### What the level is now

- **TDD order:** the 26-test file was written first against the planned exports, run to failure, then the component was rewritten to satisfy it. Mechanism pins include a ban list ('not reached', 'blocked before reaching', 'request never reaches') across all frames/probes/scenarios, '200 OK + discard' for the simple GET, 'answers the preflight' for the middleware, and 'withhold' for the evil-origin read.
- **Visualization split into CorsPipeline.tsx** (client card with browser/terminal chrome and console strip, API card with optional rack-cors strip, request/response connector pair using a rotate-180 FlowConnector for the return leg) per the components-in-own-file rule.
