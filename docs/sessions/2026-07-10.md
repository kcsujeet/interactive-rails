## Session 3: L55 deep pass (Modular Monolith mechanism-honest redesign)

### Why (the P0 from the teaching audit)

The old L55 taught Packwerk as a runtime firewall: reward scenarios fired "requests" that got "blocked" mid-flight, which is not how static analysis works. It also skipped the binstub install step, taught `enforce_privacy` as core Packwerk (it moved to the packwerk-extensions gem), and led with artifact pain ("no boundaries") instead of customer damage.

### Design decisions (design-level worksheet outputs)

- **Narrative (Step 1).** Problem in one sentence: with 12 engineers and no internal boundaries, invisible cross-domain references turn safe internal changes into production incidents, and no ownership map means incidents start with a manhunt. Actor table: Customer (receipts, refunds, stock), CI pipeline (green through everything; checks behavior, not boundaries), Billing code, Notifications code, Orders code, Inventory code. Same six-node topology in observe and reward; only state changes, and that contrast is the lesson.
- **Canonical purpose (Step 0.5, doc-verified 2026-07-10).** Packwerk README: static analysis of constant references; install is `gem 'packwerk'` + `bundle binstub packwerk` + `bin/packwerk init`; "We recommend running bin/packwerk check in your CI pipeline." USAGE.md: a package is any folder with a package.yml (path = name); `enforce_dependencies: true|strict` + `dependencies:` list; recommended public-constants namespace. Privacy checking moved to packwerk-extensions, NOT core, so the level stops teaching `enforce_privacy`.
- **Smell test (Step 0.45, written):** the player fires "Rename a notifications helper (their own code)" and watches CI stay green, the deploy ship, billing's InvoiceSender crash with NoMethodError, and customers stop receiving receipts while the team that made the change has no idea. Reaction: "a safe change broke someone else's feature in production and nothing caught it; I have to make that impossible." PASS: damage is customer-concrete, and the villain (the invisible reference) is named.
- **Probe playthrough (written, per frame):** (1) internal-rename: notifications amber RENAME -> CI green MERGED, deploy edge flows green -> billing red 500 "InvoiceSender: NoMethodError" with the hidden billing->notifications edge lighting red -> customers red NO RECEIPTS, support tickets edge. (2) circular-hotfix: inventory amber FIX + customers red OVERSOLD -> both circular edges light red, orders red REVIEW? "fix drags orders code in" -> inventory STALLED, customers DAY 2. (3) incident-owner: billing red 2:04am, refunds stuck -> "payment.rb: 12 authors, 0 owners" WHO? badge, CI sublabel "no ownership map anywhere" -> 2:44am debugging starts, customers 40 MIN. Three distinct shapes (deploy-then-crash, circular stall, pager roulette), test-enforced distinct.
- **Mechanism-honest reward.** Scenarios are PULL REQUESTS hitting the CI gate, not runtime requests: `method: 'PR'`. "Blocked" = the PR fails in CI before merge (left-panel legend says exactly that). The blocked rename replays with the CI node going red "boundary check: billing references this constant" and the deploy edge reversed "PR blocked before merge"; production never sees the break. Paired scenarios for all three probes plus a reward-only strict-mode scenario (legacy violations grandfathered; no NEW ones can land).

- **Build = 7 steps, doc-faithful:** `bundle add packwerk && bundle binstub packwerk` (terminal), `bin/packwerk init` (terminal), carve packs/billing (folder + package.yml, git mv), public API namespace (`Notifications::Public::SendReceipt`), declare dependencies (short list beats all-on-all and none), CI gate (`bin/packwerk check` in ci.yml; wrong options are manual-releases and pre-commit hooks), CODEOWNERS (per-pack teams beat one-CTO and git-blame).
