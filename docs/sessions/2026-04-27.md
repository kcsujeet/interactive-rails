## Batch 26: pedagogy guardrail (rule + test helper)

### Why

While implementing L49 Batch B locally, the observe phase shipped
with three probes that update the left-panel discovery checklist
but do NOT visibly mutate the center-panel PipelineFlow. The
player fires a probe, reads the response in the terminal, and
sees nothing change in the diagram itself. This violates the
three-phase pedagogy rule: the observe phase exists so the player
ACTIVELY discovers the problem through visible feedback in the
visualization.

The structural tests I wrote for L49 (probe-discovery 1:1,
probe-scenario coverage, scenario uniqueness, code preview
boundaries, production-safe-default invariants) all passed. None
of them looked at the question "does anything change on screen
when the player fires a probe?".

The deeper guidance for this check already exists in the
`audit-level` skill ("Step 3: Probe-by-probe playthrough" and
the "Gate Check: Does the Observe Phase Teach the Concept?"
section) and in the `design-level` skill ("Probe
Differentiation" and "The Literal Screen Test"). Both skills are
extensive and correct. The failure was that they are
invoke-on-demand: I built L49 without consulting them,
structural tests passed, and the violation was caught only when
the user noticed nothing happens on screen.

### Layered fix

Two interventions, neither replacing the existing skills.

1. **Always-loaded rule file** at `.agents/rules/pedagogy.md`,
   path-scoped to `**/Level*.tsx`, `**/content.ts`,
   `**/data/*.ts`. Loads automatically when any level file is
   touched. Intentionally short (~70 lines): states the hard gate
   ("write down what the player literally sees in the center panel
   for every probe; if your honest answer is 'the probe terminal
   shows response lines', the level is broken"), names the
   required data structure (probe-keyed visualization state map),
   and points back to the deeper skills. The L49 failure is the
   worked example.
2. **CI test helper** at `src/lib/testing/probe-pedagogy.ts`
   exporting two functions:
   - `expectEveryProbeDrivesVisualChange<TState>({ probes,
     probeStateMap, validate })` -- fails if any probe lacks an
     entry or has an entry the validator rejects as "no visible
     delta". The validator is level-specific so it can know the
     state shape.
   - `expectEveryProbeDrivesDistinctChange<TState>({ probes,
     probeStateMap, serialize })` -- fails if two probes produce
     identical visual state. Mirrors the design-level skill's
     "Probe Differentiation" requirement.

The pair is belt-and-suspenders: rule catches at design time, the
helper catches the regression at commit time.

### What this does NOT add

- No new skill. The existing `audit-level` and `design-level`
  skills already document the playthrough requirement deeply and
  correctly. Duplicating that content in a new skill would just
  add reading-and-recalling load.
- No PreToolUse hook. Hooks fire mechanically on every tool
  invocation; using one for a judgment check ("did you do the
  playthrough?") would be noise. Rules + tests already cover the
  right moments.

### Files added

- `.agents/rules/pedagogy.md`
- `src/lib/testing/probe-pedagogy.ts`

### Visual richness (follow-up)

The first iteration of this batch wired up the data correctly
(per-probe stage overrides, activeConnections, distinct visual
state per probe), but the user pointed out the actual rendered
result was still subdued: stage variant changes only modified
border colour, badges pulsed but the rest of the node was static,
and edges had no dot animation at all because connections were
never given a `dots:` prop.

Fixes applied:

1. **PipelineFlow gained a `'critical'` variant** (and the type
   union for `dots:` gained a `'danger'` preset). The new variant
   maps to FlowNode's `status: 'critical'`, which gives the whole
   card `animate-pulse border-destructive bg-destructive/10` and
   the header status dot `animate-ping` (radiating ripples). This
   is the most visceral animation the FlowNode primitive offers
   and was previously inaccessible from PipelineFlow.
2. **A new `PIPELINE_DOTS_DANGER` preset** (4 red dots, staggered)
   for use on broken-state edges. `'danger'` joined `'mixed'` and
   `'clean'` as a string preset accepted by the `dots:` prop and
   resolved by `resolveDots`.
3. **`OBSERVE_CONNECTIONS` now sets `dots: 'danger'`** on every
   edge. Red dots flow continuously through the broken pipeline,
   communicating "data is moving but every request is at risk."
4. **`REWARD_CONNECTIONS` now sets `dots: 'clean'`** on every
   edge. Green dots flow continuously through the working
   pipeline.
5. **`PROBE_OBSERVE_OVERRIDES` upgraded** the most-affected nodes
   from `variant: 'danger'` to `variant: 'critical'`. When a probe
   fires, the Flag Gate and the New Processor visibly pulse with a
   red-tinted background and ripple effect, not just a colour
   shift on the border.
6. **Base `OBSERVE_STAGES` made the FlagGate `'critical'` from the
   start.** Even before any probe fires, the missing kill switch
   is visibly broken (whole-card pulse + "MISSING" badge). This
   communicates the level's headline problem at a glance.
7. **`inspectedStages` tracking added to the orchestrator.** The
   pulsing "?" indicators were noisy alongside the new critical
   pulses. Clicking a stage now marks it `inspected: true`, which
   hides its "?" indicator the way the L11 pattern does.

After these changes, the player's literal screen experience for
each probe has multiple animated layers (whole-card pulse, badge
pulse, header dot ping, edge dot flow), not just text changes.

The pre-existing pedagogy tests still pass: the validator checked
for badge/sublabel/variant deltas, all of which still hold. The
"flag-gate is inactive" test became "flag-gate is critical" to
match the new base state.

## Batch 28: L49 visualization quality follow-ups (driven by user feedback)

After Batch 27 landed locally, the user pointed out a series of
remaining pedagogy / UX issues. Each was fixed in turn, and each
became a new section in `.agents/rules/pedagogy.md`. The journey
also surfaced a research-discipline lesson worth its own rule.

### Issues found and fixed

1. **Idle hum on initial load.** PipelineFlow's `activeConnections`
   defaults to `undefined` which puts edges into continuous "idle"
   animation. Before any probe fires, the player was seeing dots
   flow through the diagram, implying data was moving. → orchestrator
   passes `[]` (dormant) until a probe fires. New rule: "The dormant-
   edges default."

2. **Marketing probe was silent.** The reasoning "no request fires
   for this probe -- it's a scheduling gap" produced
   `activeConnections=[]`, which meant the probe terminal showed
   text but the visualization didn't move. → reframed as marketing
   firing a toggle attempt that reaches the gate but stops there.
   Upstream edges animate, downstream stays dormant. New rule:
   "Every probe needs an animated dimension."

3. **Statement-shaped probe labels.** Two of three labels were
   descriptions ("Marketing wants...", "Vendor integration starts
   misbehaving...") rather than verb-led actions. → rewritten as
   actions ("Flip launch toggle at Tuesday 9:00am sharp", "Hit kill
   switch during a vendor outage"). Matching reward scenario labels
   updated for parity. New rule: "Probe labels are verb-led actions,
   not statements."

4. **Legacy Payment Processor was a floating node.** Observe phase
   showed the Legacy node disconnected from the rest of the graph
   because OBSERVE_CONNECTIONS had no edge to it. The original
   structural test even *asserted* this absence. → keep the edge
   structurally present but omit `dots:` so the line is static.
   Anchors the node, communicates "no traffic flows here." Observe
   stage positions also aligned with reward (Y-shaped fork in both
   phases) so the layout doesn't shift between phases. New rules:
   "No floating nodes" and "Same layout across phases."

5. **Visualization felt subdued ("just text changes").** The
   per-probe overrides only changed border colour via `variant`;
   no whole-card pulse. PipelineFlow's edge connections never set
   `dots:` so no dots flowed at all. → introduced the `'critical'`
   variant on PipelineFlow (maps to FlowNode's full-card pulse + dot
   ping), added a `PIPELINE_DOTS_DANGER` preset (4 red dots), set
   `dots:` on every connection in OBSERVE_CONNECTIONS / REWARD_CONNECTIONS,
   and upgraded probe overrides to use `'critical'` on the most-
   affected nodes. The base OBSERVE_STAGES also marks the FlagGate
   `'critical'` so the broken state is visibly broken before any
   probe fires. New rule: "Visual richness requirements" with a
   variant cheat-sheet table.

6. **Active-mode dots were invisible (only the begin=0 dot played).**
   PipelineFlow's active mode override changed `dur` to `0.8s` but
   kept the original negative `begin` values used to stagger the
   indefinite-loop preset. With `begin=-2.7s, dur=0.8s, repeatCount=1`
   the animation ended 1.9s before the element mounted -- never
   played. → in active mode, also override `begin` to a positive
   cascade (`0s, 0.15s, 0.3s, 0.45s`) so every dot has a future
   window to play.

7. **Re-firing the same probe didn't replay the animation.** After
   the first single-pass burst, subsequent fires of the same probe
   were silent. SVG `<animateMotion repeatCount="1">` plays once
   then freezes; only the SVG DOM's `beginElement()` API restarts
   it (per the SMIL spec). React Flow's edge memoization defeats
   key-change tricks and prop-toggle tricks alike. After two failed
   fixes (tick-into-id, dormant-toggle), the actual fix was
   discovered by reading the [MDN beginElement docs](https://developer.mozilla.org/en-US/docs/Web/API/SVGAnimationElement/beginElement)
   and the [SMIL spec](https://svgwg.org/specs/animations/):
   set `begin="indefinite"` on each `<animateMotion>`, hold a ref,
   call `el.beginElement()` on each fire with the original begin
   parsed as a setTimeout delay for stagger. The fix lives in
   `AnimatedDots` (FlowDiagram.tsx) via a new `restartTick?: number`
   prop, plumbed through `PipelineFlow` → `PipelineFlowEdge` → the
   edge data. New rule: "Re-firing the same probe must restart the
   animation," with the canonical pattern documented and the failed
   approaches called out.

### Process lesson (separate rule)

The re-fire issue burned three rounds because the first two
attempts were guesses. The pedagogy rule now has a "Research before
guessing on browser/library quirks" section with:

- Concrete signals that should trigger a search instead of a guess
  ("works first time but not subsequent times" → animation
  lifecycle quirks; "state changes but visual doesn't update" →
  memoization or reconciliation behaviour; "the docs don't say
  either way, so I'll try X" → you are about to guess).
- The expectation that fixes for browser/library quirks cite their
  sources.
- The cost calibration: a 5-minute search up front beats two
  iterations of debugging plus a "still not working" from the user.
