## Session 1d: Gate Zone Content Behind Flow Phase

### What Changed

**File:** `frontend/src/features/act2-users-security/components/Level15CORS.tsx`

Zone 1 (CORS gate) and Zone 2 (Rails API) were showing their result text ("No headers sent", "BLOCKED", "not reached", "200 OK") immediately when a probe fired, before the flow animation reached them. This is because `probeState` was derived from `lastProbeId` which updates instantly.

**Fix:** Added two derived flags that gate content display behind the flow reaching each zone:
- `gateRevealed = probeState && (flowPhase >= 2 || flowPhase === -1)` for Zone 1
- `apiRevealed = probeState && (flowPhase >= 4 || flowPhase === -1)` for Zone 2

Replaced all `probeState` references in Zone 1's styling/content with `gateRevealed`, and in Zone 2 with `apiRevealed`. Now zones only show their result state after the animation reaches them (or after it completes).

## Session 1e: Defer Discovery Until Animation Ends

### What Changed

**File:** `frontend/src/features/act2-users-security/components/Level15CORS.tsx`

The "Build the Fix" button was appearing mid-animation because `discoveryGating.discover()` was called immediately in `handleProbe`, which could unlock the button before the flow finished.

## Session 5: Hub-and-Spoke Pipeline Layout for L5-L8

### What Changed

Replaced the linear pipeline layout (Request -> Router -> Controller -> Model -> Response) with an accurate hub-and-spoke layout where Controller is the central orchestrator and Model/Database/Serializer are its bidirectional dependencies.

**Layout:**
```
                 Serializer (L7+)
                    ^  |
                    |  v
Request -> Router -> Controller -> Response
                    ^  |
                    |  v
                   Model (L5+)
                    ^  |
                    |  v
                  Database (L5+)
```

#### PipelineFlow.tsx changes:
- Added `position?: { x: number; y: number }` to `PipelineStage` for explicit node positioning
- Added `sourceHandle?`, `targetHandle?`, `bidirectional?` to `PipelineConnection` for multi-directional edges
- Added `activeConnections?: string[]` prop for sequential edge animation control (undefined = idle, [] = dormant, [...ids] = active)
- Replaced 2 generic handles with 8 named handles (4 sides x 2 types: source + target) on every node
- Updated `buildNodes` to use explicit positions while auto-positioning unpositioned stages horizontally
- Updated `buildEdges` to expand bidirectional connections into forward + return edges, with handle ID resolution
- Return edges render with smaller dots (r=6 vs r=8) and lower opacity (0.6), no base line (shares forward edge's line)
- Edge animation modes: idle (repeatCount indefinite), active (repeatCount 1, fast dur), dormant (no dots)
- Added `fitView({ padding: 0.15 })` to prevent satellite nodes from being clipped

#### Level updates (L5, L6, L7, L8):
- All four levels now use hub layout with `HUB_POS` constants
- Main chain (Request, Router, Controller, Response) is auto-positioned horizontally
- Model at `{ x: 500, y: 180 }` below Controller, bidirectional
- Database at `{ x: 500, y: 360 }` below Model, bidirectional
- Serializer at `{ x: 500, y: -180 }` above Controller (L7+), bidirectional
- Progressive disclosure: L5-L6 show Model + Database, L7-L8 add Serializer
- Both observe and reward stages updated with positions

#### audit-level SKILL.md:
- Added "Hub-and-spoke layout for MVC architecture levels" subsection with layout diagram, position constants, and checklist
- Added "Sequential edge animation" subsection documenting the `activeConnections` prop API and flow sequence pattern

#### Follow-up fixes

**Symmetric bidirectional edges:**
Forward and return edges overlapped on the exact same path. Fixed with symmetric offset: forward edge shifts -14px left, return edge shifts +14px right (via `LANE_OFFSET`). Added `isBidirectional` flag to forward edges so only bidirectional pairs get offset (non-bidirectional horizontal edges stay centered). Return edge renders with dashed line (`strokeDasharray="4 4"`). Both directions use same dot size (r=8).

**Satellite node state:**
