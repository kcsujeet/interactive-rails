## Session 1: L26 Visualization Redesign + Audit Skill Update

User feedback on L26: the ScanBar (progress bar) "tells me nothing about indexing." Redesigned the visualization to show the mechanism, not just the metric. Then codified this lesson as a new principle in the audit-level skill.

### Changes

**Level26Indexing.tsx (visualization redesign):**
- Replaced `ScanBar` (progress bar) with `TableRowGrid`: 100 small blocks in a 20-column grid representing database rows
- Seq Scan animation: red wave sweeps through all 100 blocks left-to-right in 14 steps over ANIMATION_DURATION_MS, then match block(s) turn green
- Index Scan animation: `IndexLookupCard` appears showing B-tree entries with highlighted lookup, then only match block(s) turn green instantly (no red wave)

- Changed state from `revealedLanes: Set` to per-lane `laneProgress: Record<string, number>` (0 to GRID_SIZE) for granular scan animation
- Added `rewardProgress`/`rewardDone` separate state for reward phase
- Split animation into `runSeqScanAnimation` (14-step red wave) and `runIndexScanAnimation` (400ms pause then instant match)
- Grid match positions: email=[73], fk=[12,37,54,71,89], composite=[every other block]

**Audit skill (`.claude/skills/audit-level/SKILL.md`):**
- Added new section: "Visualization must show the mechanism, not just the metric (non-negotiable)"
- Full case study: L26 ScanBar (metric-based, failed) vs TableRowGrid + IndexLookupCard (mechanism-based, works)
- 4-item checklist: can player see what system is doing, does viz look structurally different between problem/solution, could player explain the mechanism, are progress bars the primary element
- Rule of thumb: if the visualization could be replaced by a text label without losing information, it's a metric
- Added L26 as a reference implementation in the custom visualization list at the top

### Architecture Notes

- Key principle: "mechanism vs metric." A progress bar showing "10,000 rows scanned" is a metric. A grid of blocks turning red is a mechanism. Numbers tell facts; mechanisms build mental models.
- The observe/reward phases must look structurally different. L26 observe has NoIndexBanner + red wave. L26 reward has IndexLookupCard + single green block. The new visual element (the index card) IS what the player built.
- This principle generalizes: every Type 3 visualization should show objects (rows, queries, cache entries, connections) being manipulated, not numbers about those objects.

## Session 6: Audit Skill - Gate Check for Visualization Design

User identified that L27's observe phase has no real visualization: just a ProbeTerminal showing SQL text logs. The audit caught 18 surface-level issues but missed the fundamental design flaw. Root cause analysis revealed the audit was treated as a compliance checklist (checking boxes) rather than a design review (evaluating whether the visualization teaches the concept).

### Changes

**Audit skill (`.claude/skills/audit-level/SKILL.md`):**
- Added new top-level section: "Gate Check: Does the Observe Phase Teach the Concept? (EVALUATE FIRST)"
- Positioned between "Step 0: Read Official Documentation" and "## Checklist" so it runs BEFORE any structural compliance checks
- **Two-step process:** First identify the observe type (1-4), then apply the design check specific to that type
- Type-specific design checks:
  - Type 1 (no observe): passes automatically
  - Type 2 (static intro): "Does the annotated code make the structural problem self-evident?"
  - Type 3 (custom visualization): 3 sub-checks about visual components, animated objects, and probe-driven reactions
  - Type 4 (PipelineFlow): 3 sub-checks about node variant changes, sequential edge animation, and StageInspector content
- Explicit instruction: if any sub-check fails for the level's type, do NOT proceed with the rest of the audit
- Full L27 case study documenting the three reasons the failure was missed
- ProbeTerminal vs Visualization distinction scoped to Types 3 and 4 only (Types 1 and 2 don't use ProbeTerminal)
- ASCII diagrams showing correct vs broken architecture
- Updated "mechanism vs metric" checklist: added terminal-log-specific failure modes
- **Second revision:** User pointed out the original gate check was written only for Type 3, ignoring Types 1/2/4. Rewrote to be type-aware with a lookup table and per-type checks.

### Architecture Notes

- The audit process has a hierarchy of importance. Visualization design (does it teach the concept?) is the foundation. Everything else (animation locking, color contrast, shared components, step quality) is predicated on having a real visualization to check. A perfectly compliant level with no visualization is worse than a non-compliant level with a good visualization.
- The gate check must be type-aware. What "teaches the concept" means is different for each observe type. A static annotated code display is correct for Type 2 but wrong for Type 3. A ProbeTerminal-only observe is wrong for Type 3 but ProbeTerminal does not even apply to Types 1 and 2.
- ProbeTerminal is a discovery TOOL, not a visualization. It belongs BELOW a visualization and drives it via `onProbe` callbacks. This applies to Types 3 and 4 only.
- This lesson generalizes: when auditing, always evaluate the design intent before the implementation details. Fixing 18 surface issues on a fundamentally broken design is wasted work.

### Architecture Notes

This is a mechanism (you see the queries happening), not a metric (you read a number).
