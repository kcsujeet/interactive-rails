# Session Log: 2026-03-11

## Session 1: L26 Visualization Redesign + Audit Skill Update

User feedback on L26: the ScanBar (progress bar) "tells me nothing about indexing." Redesigned the visualization to show the mechanism, not just the metric. Then codified this lesson as a new principle in the audit-level skill.

### Changes

**Level26Indexing.tsx (visualization redesign):**
- Replaced `ScanBar` (progress bar) with `TableRowGrid`: 100 small blocks in a 20-column grid representing database rows
- Seq Scan animation: red wave sweeps through all 100 blocks left-to-right in 14 steps over ANIMATION_DURATION_MS, then match block(s) turn green
- Index Scan animation: `IndexLookupCard` appears showing B-tree entries with highlighted lookup, then only match block(s) turn green instantly (no red wave)
- Added `IndexLookupCard` component: shows sorted B-tree index structure (e.g., `alice@example.com -> row #4231`) with arrow on the highlighted entry
- Added `NoIndexBanner` component: red banner showing "No index on {table}, Reading every row (Sequential Scan)"
- Changed state from `revealedLanes: Set` to per-lane `laneProgress: Record<string, number>` (0 to GRID_SIZE) for granular scan animation
- Added `rewardProgress`/`rewardDone` separate state for reward phase
- Split animation into `runSeqScanAnimation` (14-step red wave) and `runIndexScanAnimation` (400ms pause then instant match)
- Grid match positions: email=[73], fk=[12,37,54,71,89], composite=[every other block]

**Audit skill (`.claude/skills/audit-level/SKILL.md`):**
- Added new section: "Visualization must show the mechanism, not just the metric (non-negotiable)"
- Full case study: L26 ScanBar (metric-based, failed) vs TableRowGrid + IndexLookupCard (mechanism-based, works)
- 4-item checklist: can player see what system is doing, does viz look structurally different between problem/solution, could player explain the mechanism, are progress bars the primary element
- Rule of thumb: if the visualization could be replaced by a text label without losing information, it's a metric
- Added L26 as a reference implementation in the custom visualization list at the top

### Architecture Notes

- Key principle: "mechanism vs metric." A progress bar showing "10,000 rows scanned" is a metric. A grid of blocks turning red is a mechanism. Numbers tell facts; mechanisms build mental models.
- The observe/reward phases must look structurally different. L26 observe has NoIndexBanner + red wave. L26 reward has IndexLookupCard + single green block. The new visual element (the index card) IS what the player built.
- This principle generalizes: every Type 3 visualization should show objects (rows, queries, cache entries, connections) being manipulated, not numbers about those objects.

## Session 2: L26 Scroll Fix + Expected Scan Amber Colors

Two UX bugs fixed after testing the redesigned L26 visualization.

### Changes

**Scroll overflow fix (Level26Indexing.tsx):**
- Probe buttons in the center panel were hidden below the viewport because nested flex containers lacked height constraints
- Added `min-h-0` to parent flex container, observe phase wrapper, observe scroll div, and reward phase wrapper
- This is the standard pattern for enabling overflow scrolling in nested flex layouts

**Expected seq scan amber colors (Level26Indexing.tsx):**
- "Admin: list all users" had a green button (expectedResult: 'allowed') but the visualization showed red blocks and "SLOW" badge, which was confusing
- Added `expected?: boolean` flag to `ScanResult` interface to distinguish problematic seq scans from acceptable ones
- Set `expected: true` on the `admin-all-users` reward scenario (full table scan is correct when loading all rows with no WHERE clause)
- `TableRowGrid`: amber block colors (`bg-amber-300/80 dark:bg-amber-500/50`) and amber text instead of red when `expected` is true
- `QueryPlanLane`: amber borders, badge text ("EXPECTED"), and timing text when scan is expected
- `NoIndexBanner`: amber variant with `Info` icon instead of `AlertTriangle`, text reads "Expected when loading all rows"

### Architecture Notes

- Not all sequential scans are bad. When there's no WHERE clause (e.g., `User.all` for an admin list), a full table scan IS the correct strategy. The visualization should reflect this with amber (informational) instead of red (problematic).
- The `expected` flag is set per-scenario in the reward phase data, not globally. Each scenario independently controls whether its seq scan is expected or not.

## Session 3: L26 Data Consistency Fix + Tests + Audit Skill Update

User spotted that the "Admin: list all users" reward scenario showed `SELECT * FROM users WHERE email = 'alice@example.com'` in the lane header alongside a "no WHERE clause" banner. Fixed the contradiction, wrote tests to catch similar issues, and updated the audit skill.

### Changes

**Reward lane override for admin scenario (Level26Indexing.tsx):**
- The "Admin: list all users" scenario reuses the `email` lane but has no WHERE clause. The lane header was showing `SELECT * FROM users WHERE email = 'alice@example.com'` alongside a "no WHERE clause" banner, which was contradictory.
- Added `sqlOverride` and `labelOverride` fields to reward scan data type
- `admin-all-users` now overrides SQL to `SELECT * FROM users` and label to "Admin: All Users"
- `QueryPlanLane` accepts optional `sqlOverride`/`labelOverride` props, falling back to the lane definition

**Audit skill (`.claude/skills/audit-level/SKILL.md`):**
- Added new section: "Reward scenario data must not contradict shared visualization components (non-negotiable)"
- Full L26 case study: lane SQL showing WHERE clause while banner says "no WHERE clause"
- 4-item checklist: SQL/plan consistency, label context, banner text accuracy, write automated tests
- Placed in Cross-Phase Consistency section before "Reward phase type must match"

**Tests (`__tests__/Level26Indexing.test.ts`, 35 tests):**
- **Reward data consistency**: every scenario has reward scan entry, maps to valid lane, SQL does not contradict plan text, expected scans require sqlOverride when lane has WHERE, expected scans have labelOverride
- **Response lines**: all scenarios have them, allowed index scans have green lines, blocked scenarios have red lines
- **Grid matches**: all positions within bounds, email has 1 match, fk has multiple, composite has 50%
- **Observe phase**: all lanes have data, all scans are seq scans, all read full table, probes map to valid discoveries/lanes
- **Build step quality**: correct answer never first, every wrong option has feedback, feedback never reveals answer, each step has exactly one correct answer
- **Cross-phase consistency**: observe and reward cover same lanes, seq scans become index scans in reward, probe/scenario labels match
- **Scenario uniqueness**: unique IDs, unique labels, mix of allowed/blocked

**Project rules (`.claude/rules/testing.md`):**
- New rules file scoped to `**/*.{ts,tsx}`
- Mandates tests for every level: validation logic, build step quality, data consistency, cross-phase consistency, scenario uniqueness
- Specifies `bun:test` framework, `__tests__/` directory pattern, pure logic testing (no React rendering)
- Special checks for levels with shared visualization components (lane/zone SQL contradictions, override requirements)
- Mirror data into tests rather than importing from components
