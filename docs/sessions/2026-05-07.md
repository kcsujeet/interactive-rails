**L13 redesign** (uncommitted):
Full rewrite around mass-assignment exploits:
- Pre-state: controllers use `to_unsafe_h`. Marketing has added a `featured: boolean` column on Product (admin-only homepage flag).
- Three problem-probes (all real exploits, all blocked after the build):
  - `self-promote-create`: POST with `featured: true` -> product saved as featured.
  - `self-promote-update`: PATCH with `featured: true` -> existing product flipped.
  - `compound-frame`: PATCH with `featured: true, user_id: 99` -> featured AND ownership transferred to victim.
- Build phase: replace `to_unsafe_h` with `params.expect(product: [:name, :description, :price])`, centralized into `product_params` private method, called from both create and update.
- Reward: same three exploits now blocked; clean create/update still pass.
- DISCOVERY_DEFS, PROBES, PROBE_DISCOVERY_MAP, PROBE_PIPELINE_MAP, STAGE_INSPECTOR_MAP, STRESS_SCENARIOS, FILTERING_OPTIONS, WHITELIST_OPTIONS, OPTION_STEP_CONFIG, STEP_DEFS, PROBE_ACTIVE_CONNECTIONS, SCENARIO_ACTIVE_CONNECTIONS, getCodeFiles, header comment, and content/level-13-strong-params.ts (trigger, problem, codeExample, learningContent, hint) all rewritten.

### Why

The previous session's L13 design ran through three failure modes:
1. Pre-baked `params.require/permit` at L7 (earned-abstraction violation).
2. Stripped to explicit-field extraction (lesson-survival violation: form is mass-assignment-safe by construction; the bug L13 fixes does not exist in the before-state).
3. Initial fix proposal scoped `to_unsafe_h` to L13's before-state only — would have created a localized regression from L7-L12's safer form.

A separate misstep overgeneralized "every concept has a naive precursor" — false. Authentication (L9) is existence-axis with no precursor. The corrected framing: **cumulative-patterns operates on two orthogonal axes (existence vs form); only form-axis levels have precursors, and those precursors must satisfy both earned-abstraction (no future API) AND lesson-survival (the bug the future level fixes still exists).**

Per the [Rails Action Controller guide](https://guides.rubyonrails.org/action_controller_overview.html): *"With Action Controller Strong Parameters, parameters cannot be used in Active Model mass assignments until they have been explicitly permitted."* That is the canonical purpose; the L13 redesign is built around it.

### Skill rule extension: cross-verify against myapp (2026-05-09)

The L13 chronology surfaced a missing rule: src/ and myapp can fall out of sync if either is updated without the other. For ~24 hours after the L13 redesign committed, src/ taught `params.expect(product: [:name, :description, :price])` while myapp's level-13 tag still had `params.require(:product).permit(...)`. A player following the curriculum end-to-end with `git checkout level-13` in myapp would have seen a different controller than the game promised.

Codified in two skills:
- **audit-level/SKILL.md** new section "Step 0.5: Cross-verify against the myapp project" (NON-NEGOTIABLE). Lists what to cross-check (`getCodeFiles` reward snapshot, stage inspector code, probe response lines, terminal output, code preview transitions) and the canonical comparison: `git show level-N:<path>` in myapp vs. the simulated content in src/. Source-of-truth flow: official docs → real command in myapp → level data, never the reverse.
- **design-level/SKILL.md** new section "Step 0.4: Anchor to the myapp project" (NON-NEGOTIABLE). Tells the designer to check out the previous level's myapp tag before designing any code preview / generator / response line; run real commands in myapp and mirror the actual output into the level; never invent from memory. Order of operations: official docs (Step 0.5) → real myapp command (Step 0.4) → level data.

Both sections carry the L13 case study with the explicit rule that triggered the misalignment.
