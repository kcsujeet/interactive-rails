# Session Log: 2026-03-17

## L34 Visualization Redesign

User feedback: the dual timeline lanes visualization was confusing. Both Thread A and Thread B showed identical green READ/COMPUTE/WRITE boxes with "COMMITTED" status, making it impossible to see the race condition. No actual values were shown in the steps.

### Redesign: Race Timeline Table

Replaced the abstract `TimelineLane` component with a `RaceTimeline` table that shows actual values at each moment in time:

**Before (confusing):** Two horizontal swim lanes with generic READ/COMPUTE/WRITE boxes, all turning green. An "Account Row $50" card above was disconnected from the lanes.

**After (clear):** A 3-column grid (Thread A | Balance | Thread B) where rows appear one by one, each showing the exact value being read, computed, or written. The race condition becomes visible when you see both threads reading "$100" simultaneously, then Thread B's "$50 SAVE (overwrites)" turning the balance red while Thread A's "$70" disappears.

**Data-driven timelines:**
- `OBSERVE_CONCURRENT_ROWS`: 4 rows showing balance race condition ($100 -> $70 -> $50 LOST!)
- `OBSERVE_STALE_ROWS`: 4 rows showing profile overwrite (Alice -> Carol -> Bob LOST!)
- `REWARD_ALLOWED_ROWS`: 4 rows showing serialized access with locks ($100 -> $70 -> $20 correct)
- `REWARD_BLOCKED_ROWS`: 3 rows showing rejection with rollback ($100 -> CHECK FAILS -> $100 safe)

**State simplification:** Replaced 7 state variables (`threadA`, `threadB`, `accountBalance`, `rewardThreadA`, `rewardThreadB`, `rewardBalance`, `rewardLocked`) with 4 (`observeRows`, `observeVisible`, `rewardRows`, `rewardVisible`). Animation is now just incrementing `visibleCount` with `setTimeout`.

**Empty state:** Before any probe fires, a dashed placeholder with "Fire a probe to see the race condition unfold step by step" gives clear affordance.
