# Session Log: 2026-03-19

## Session 1: L35 Active Storage - Redesign Observe Phase Visualization

### What Changed

**Problem:** L35 had three probes (upload, download, list) that all played the same generic animation: Client lights up -> App Server lights up -> S3 lights up. Each probe tests a completely different operation but the visualization didn't differentiate between them. A player firing "Download user avatar" saw "Sending file..." and "Stored", which makes no sense for a download.

**Solution:** Replaced the generic `flowPhase` animation with L34's frame-based `AnimationFrame` pattern. Each probe now plays a distinct multi-frame animation:

1. **Upload 5MB photo**: Data flows left-to-right (Client -> App -> S3). Memory gauge on App Server fills from 45MB to 95MB (red). Warning explains memory buffering.

2. **Download user avatar**: Data flows **right-to-left** (S3 -> App -> Client). FlowConnector dots travel left via `-scale-x-100` CSS transform. App Server shows "Worker BLOCKED 3s". Completely different visual from upload.

3. **List users with avatars**: No directional flow. Instead, App Server zone accumulates entries ("User 1... User 2... x25 users, no variants!"). Bandwidth counter shows "125MB served (should be 375KB)". Visual emphasis on repetition and volume.

**Reward phase** also got per-scenario animations:
- Direct upload: Client gets presigned URL from App, then uploads directly to S3 (green dots, memory unchanged)
- Serve variant: 302 redirect from App, S3 serves 15KB thumbnail to Client (right-to-left green dots)
- Blocked scenarios: Specific rejection frames (content type validation, file size contract)

### Technical Details

- Added types: `ZoneFlash`, `ZoneState`, `ConnectorVizState`, `AnimationFrame`
- Added frame sequences: `UPLOAD_PROBE_FRAMES`, `DOWNLOAD_PROBE_FRAMES`, `LIST_PROBE_FRAMES`, plus 6 reward frame sequences
- Added `MemoryGauge` component showing memory bar on App Server zone
- Added `applyFrame()` and `runAnimation()` helpers (same pattern as L34)
- Replaced `flowPhase` state with `clientState`, `appState`, `s3State`, `connAState`, `connBState`, `memoryMB`, `warningMessage`, `bandwidthLabel`, `vizAnimating`
- Fixed existing bugs: `FlowConnector` was receiving invalid `direction="right"` (should be `"horizontal"`) and invalid `dotColor="green"` (should be Tailwind bg class)
- Used `-scale-x-100` on FlowConnector wrapper div to reverse dot direction for download/serve-variant animations

### Build/Test Results
- `bun run build`: passes, no type errors
- `bun test Level35ActiveStorage.test.ts`: 47/47 pass (test data unchanged, only visualization code changed)

## Follow-up: Fix Observe Phase Narrative Inconsistency

**Problem:** The observe phase code preview showed `user.avatar.attach(@file)` and `send_data user.avatar.download`, both Active Storage APIs. But the build phase step 0 says "Active Storage not installed yet" and asks the player to install it. If AS isn't installed, those APIs wouldn't exist.

**Fix:** Changed observe phase code preview to show manual file handling (no Active Storage):
- Service: `File.binwrite(path, @file.read)` instead of `user.avatar.attach(@file)`
- Controller: `send_file user.avatar_path` instead of `send_data user.avatar.download`
- Build step 0 preview: Updated comment to "No Active Storage yet. Manual file handling."
- Download probe response line: `send_file user.avatar_path` instead of `send_data @user.avatar.download`
- Download animation frame: `send_file avatar_path` instead of `send_data avatar.download`
- Updated test mirrors to match new probe response text

Now the narrative is consistent: observe shows naive manual handling, build upgrades to Active Storage.

Also fixed `content.ts` `problem.codeExample` (the briefing screen) which still showed `user.avatar.attach(@file)` and `send_data user.avatar.download`. Updated to match the manual file handling approach (`File.binwrite`, `send_file user.avatar_path`).
