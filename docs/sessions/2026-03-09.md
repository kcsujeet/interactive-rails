## Session 6: Fix Auto-Fire in StressTestPanel

Auto-fire had two bugs: it looped forever (cycling through scenarios indefinitely) and it never triggered level animations (bypassed the `onFire` handler, calling `addResult` directly).

### Root causes

1. **Infinite loop:** `autoFireIndexRef.current % scenarios.length` wraps around forever. Should stop after one cycle.
2. **No animations:** The hook called `addResult(scenarioId)` internally during auto-fire, which only updated counters/results. It never called the level's `onFire` handler (which is `handleFireScenario`), so no `runFlow()`, no `setLastRewardResult()`, no pipeline updates.

### Fix

**`useStressTest.ts` hook:**
- Changed `toggleAutoFire` signature from `() => void` to `(onFire: (scenarioId: string) => void) => void`
- Auto-fire now delegates to the `onFire` callback (same handler as manual fire) instead of calling `addResult` directly
- Switched from `setInterval` to chained `setTimeout` for cleaner scheduling
- Stops after cycling through all scenarios once (no modulo wrap-around)
- Speed escalation: starts at 1200ms, decreases by 100ms per tick, minimum 600ms
- Stops with a 500ms delay after final scenario fires

**`StressTestPanel.tsx`:**
- Updated `onToggleAutoFire` prop type to `(onFire: (scenarioId: string) => void) => void`
- Auto button `onClick` now passes `onFire` through: `() => onToggleAutoFire(onFire)`
- This means auto-fire calls the exact same `onFire` handler as clicking scenario buttons manually

**`Level15CORS.tsx`:**
- Fixed wrapper `() => stressTest.toggleAutoFire()` that swallowed the `onFire` argument. Changed to `stressTest.toggleAutoFire` (direct reference)

**All other levels:** No changes needed. They already pass `stressTest.toggleAutoFire` directly, which now has the correct `(onFire) => void` signature that matches the updated prop type.

## Session 6c: Add Animation Locking to L23

Applied the animation locking pattern to L23's ProbeTerminal and StressTestPanel.

### Changes

**`Level23N1Problem.tsx`:**
- Added `isAnimating` state (boolean) alongside `flowPhase`. This is needed because L23's `flowPhase` persists at 4 after animation (visual state stays until next probe), so `flowPhase !== -1` would permanently disable buttons.
- `runFlow()` now sets `isAnimating = true` at start, and schedules `setIsAnimating(false)` 1.5s after the final phase (phase 4 at 2400ms + 1500ms buffer for query cascade).
- ProbeTerminal: added `disabled={isAnimating}`
- StressTestPanel: added `disabled={isAnimating}`

### Why `isAnimating` instead of `flowPhase !== -1`
L23 has a unique requirement: the flow animation visual state persists at phase 4 until the next probe fires (showing the completed query flood, database panic, etc.). Other levels reset `flowPhase` to -1 after animation, so they can use `flowPhase !== -1` directly. L23 needs a separate flag because the visual state and the "controls locked" state have different lifetimes.
