## Session 6: Fix Auto-Fire in StressTestPanel

Auto-fire had two bugs: it looped forever (cycling through scenarios indefinitely) and it never triggered level animations (bypassed the `onFire` handler, calling `addResult` directly).

### Root causes

1. **Infinite loop:** `autoFireIndexRef.current % scenarios.length` wraps around forever. Should stop after one cycle.
2. **No animations:** The hook called `addResult(scenarioId)` internally during auto-fire, which only updated counters/results. It never called the level's `onFire` handler (which is `handleFireScenario`), so no `runFlow()`, no `setLastRewardResult()`, no pipeline updates.

### Fix

**`useStressTest.ts` hook:**
- Changed `toggleAutoFire` signature from `() => void` to `(onFire: (scenarioId: string) => void) => void`
- Auto-fire now delegates to the `onFire` callback (same handler as manual fire) instead of calling `addResult` directly
- Switched from `setInterval` to chained `setTimeout` for cleaner scheduling
- Stops after cycling through all scenarios once (no modulo wrap-around)
- Speed escalation: starts at 1200ms, decreases by 100ms per tick, minimum 600ms
- Stops with a 500ms delay after final scenario fires

**`StressTestPanel.tsx`:**
- Updated `onToggleAutoFire` prop type to `(onFire: (scenarioId: string) => void) => void`
- Auto button `onClick` now passes `onFire` through: `() => onToggleAutoFire(onFire)`
- This means auto-fire calls the exact same `onFire` handler as clicking scenario buttons manually

**`Level15CORS.tsx`:**
- Fixed wrapper `() => stressTest.toggleAutoFire()` that swallowed the `onFire` argument. Changed to `stressTest.toggleAutoFire` (direct reference)

**All other levels:** No changes needed. They already pass `stressTest.toggleAutoFire` directly, which now has the correct `(onFire) => void` signature that matches the updated prop type.

## Session 8b: L24 Visualization Redesign (Strategy Cards -> SQL Timeline Lanes)

The initial StrategyCard approach (session 8) was rejected: cards were just static labels with text. They didn't show HOW each strategy works. The player couldn't see the difference between 2 queries vs 1 JOIN vs 101 queries.

### New visualization: "SQL Timeline Lanes"

4 horizontal lanes (includes, preload, eager_load, joins), each showing the actual SQL query pattern as colored blocks. When a probe fires, all 4 lanes animate left-to-right with staggered reveals (300ms between lanes).

Key visual elements:
- **Query blocks**: colored rounded pills showing SQL snippets (green=works, amber=suboptimal, red=error)
- **Flood blocks**: for N+1 lanes (joins), 60 tiny 5x5px red squares burst in after the initial query block, visually making the 101-query explosion obvious without reading text
- **Result badges**: WORKS/N+1!/OK/ERROR per lane with total query count
- **Lane headers**: strategy name (clickable with `?` indicator for StageInspector) + method signature

Before any probe fires, lanes show grayed-out empty state: "Fire a probe to test this strategy."

### Probe lane data (per probe, all 4 strategies)

**Probe 1 (basic users):**
- includes: [SELECT posts] -> [SELECT users IN(...)] = 2 queries WORKS
- preload: [SELECT posts] -> [SELECT users IN(...)] = 2 queries WORKS
- eager_load: [SELECT posts LEFT JOIN users] = 1 query WORKS
- joins: [SELECT posts JOIN users] -> [100 red flood blocks] = 101 queries! N+1!

**Probe 2 (nested comments):**
- includes(comments: :user): [posts] -> [comments IN] -> [users IN] = 3 queries WORKS
- preload(comments: :user): same 3 queries WORKS
- eager_load(comments: :user): [wide JOIN] = 1 query SUBOPTIMAL (high memory)
- joins(:comments): [posts JOIN comments] -> [100 red flood] = 100+ queries! N+1!

**Probe 3 (filtered association):**
- includes(.where): [LEFT JOIN WHERE] = 1 query SUBOPTIMAL (auto-switches)
- preload(.where): [ERROR!] = cannot filter with separate queries
- eager_load(.where): [LEFT JOIN WHERE] = 1 query WORKS
- joins(.where): [JOIN WHERE] -> [50 red flood] = 51+ queries! N+1!
