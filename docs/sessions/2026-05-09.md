## L14 Testing: third redesign, "Behavior Coverage" Type 3 visualization

### What changed

Replaced both the observe and the reward visualizations on L14. Previous attempts both failed:

1. **CI/CD pipeline (Code → Build → Test Gate → Production)**: pre-baked future concepts (deploy, CI, staging, production) the curriculum doesn't introduce until L40+.
2. **Editor + Test Runner PipelineFlow**: taught dev *tools* (an app and a command) instead of teaching what testing IS.

The new design teaches that **a test is a plain-English assertion about app behavior, written down once, then checked automatically on every change**.

**Center panel (both observe and reward)**: a stack of three "Behavior" cards — one per assertion the player has built across L10-L13:

1. *Admin-only fields stay admin-only* (the L13 strong-params lesson)
2. *Only the owner can update or delete their product* (the L11 Pundit lesson)
3. *User contact info works through the encrypted column* (the L10 encryption lesson)

Each card has:
- Status icon (`?` muted = no spec yet; `✓` green = covered; `✗` red = failing)
- Plain-English title
- Mono `it "..."` snippet (the actual RSpec example description)
- Live status text ("No spec yet. No automated check exists for this rule.")

A footer banner shows **"X / 3 behaviors covered"**: pre-L14 = `0 / 3`, post-L14 = `3 / 3`, regression = `2 / 3` red.

**Probes rewritten** with outcome / regression language:
- `Re-introduce the admin-promotion bug from L13`
- `Re-introduce the unauthorized-delete bug from L11`
- `Re-introduce the email_address column rename`

**Scenarios** mirror the probes one-for-one (same labels, same behavior targets) and add a clean-refactor `Refactor a private helper method (no behavior change)` so the player can see safe edits stay green.

## L14 (Testing): drop CI/CD framing, redesign observe + reward visualization

### Why

The previous L14 visualization was a 4-node CI/CD pipeline (`Code` -> `Build` -> `Test Gate` -> `Production`) with probes labeled `Deploy: drop params.expect filter`, `Deploy: skip authorize in update`, `Deploy: rename users.email_address column`. None of those concepts (CI, build, deploy, staging, production) exist at L14. They're introduced at L40+. Per the audit-level skill's Step -1 Q4 ("Visualization vocabulary must match curriculum state") and the new "Inactive doesn't redeem pre-bake" clause, rendering future-concept nodes as dimmed/dashed makes them baffling, not honest. The fix is to remove the pipeline framing entirely.

### What

**Visualization (Type 4 PipelineFlow, two honest nodes):** `Editor` (the file the player edits) -> `Test Runner` (`bundle exec rspec` output). One edge between them. Matches the actual L14 workflow: edit a file, run rspec, get pass/fail.

- **Observe:** Editor sublabel changes per probe (e.g., "products_controller.rb (params.expect dropped)"). Test Runner stays at "0 examples, 0 failures" with `inactive` variant + `NO SPECS` badge because no specs exist yet.
- **Reward:** Editor is static. Test Runner is `default` ("6 examples, ready") before any scenario fires; flips to `active` + `SUITE GREEN` ("6 examples, 0 failures") on a clean refactor; flips to `danger` + `SUITE RED` ("6 examples, 1 failure") on any regression.

- `pundit-regression`: "Edit: remove authorize from destroy"
- `schema-regression`: "Edit: rename users.email_address to email"
- `clean-refactor`: "Edit: rename a private helper method"

**Probe stories** rewritten to "you edit the file, save, run rspec, see no signal" framing. **Scenario stories** rewritten to "you make the same edit again; this time the spec exists; rspec turns red". `method` field changed from `PUSH` -> `EDIT`; `actor` changed from `dev_alice/dev_bob/dev_charlie` -> `you`.

**Stage inspector map** rewritten for the two honest nodes. `editor` shows the actual L13 controller code (params.expect filter, authorize calls); `runner` shows the empty rspec output and explains why no specs exist.

**Content.ts:** `trigger.description`, `problem.observation`, `problem.rootCause`, `problem.codeExample` all reframed around editing locally and running rspec (no "deploy", no "ship", no "production").

**Test file:** mirrors new probe/scenario labels; new `describe` block "visualization vocabulary (curriculum state)" enforces every label starts with `Edit:` and contains no forbidden vocabulary (`Deploy`, `CI `, `staging`, `Production`, `Test Gate`, `Build`, `PUSH`, `git push`, `PROMOTED`, `HIJACKED`). Stage IDs constrained to `editor` + `runner` only.
