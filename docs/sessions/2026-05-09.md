## L14 Testing: third redesign, "Behavior Coverage" Type 3 visualization

### What changed

Replaced both the observe and the reward visualizations on L14. Previous attempts both failed:

1. **CI/CD pipeline (Code → Build → Test Gate → Production)**: pre-baked future concepts (deploy, CI, staging, production) the curriculum doesn't introduce until L40+.
2. **Editor + Test Runner PipelineFlow**: taught dev *tools* (an app and a command) instead of teaching what testing IS.

The new design teaches that **a test is a plain-English assertion about app behavior, written down once, then checked automatically on every change**.

**Center panel (both observe and reward)**: a stack of three "Behavior" cards — one per assertion the player has built across L10-L13:

1. *Admin-only fields stay admin-only* (the L13 strong-params lesson)
2. *Only the owner can update or delete their product* (the L11 Pundit lesson)
3. *User contact info works through the encrypted column* (the L10 encryption lesson)

Each card has:
- Status icon (`?` muted = no spec yet; `✓` green = covered; `✗` red = failing)
- Plain-English title
- Mono `it "..."` snippet (the actual RSpec example description)
- Live status text ("No spec yet. No automated check exists for this rule.")

A footer banner shows **"X / 3 behaviors covered"**: pre-L14 = `0 / 3`, post-L14 = `3 / 3`, regression = `2 / 3` red.

**Probes rewritten** with outcome / regression language:
- `Re-introduce the admin-promotion bug from L13`
- `Re-introduce the unauthorized-delete bug from L11`
- `Re-introduce the email_address column rename`

**Scenarios** mirror the probes one-for-one (same labels, same behavior targets) and add a clean-refactor `Refactor a private helper method (no behavior change)` so the player can see safe edits stay green.

## L15 build phase: off-by-one between caller's codePreviewStep and getCodeFiles thresholds

### What changed

User flagged via screenshot: after completing step 2 (Send the Welcome Email), the right panel did not show `app/controllers/users_controller.rb` even though the step had just modified that file. Root cause: thresholds in `data/code-files.ts` were one step ahead of the caller's `codePreviewStep` convention.

The caller (`Level15Callbacks.tsx`) follows CLAUDE.md's standard:
```ts
const codePreviewStep = stepper.isCurrentStepCompleted
  ? stepper.currentStep
  : stepper.currentStep - 1;
```
So when step 2 has just been completed, `codePreviewStep = 2` (the 0-indexed step index of the last completed step).

But `code-files.ts` had:
```ts
if (furthestStep >= 3) { /* show UsersController */ }   // wrong: should be >= 2
if (furthestStep >= 4) { /* show ProductsController */ } // wrong: should be >= 3
```

Each artifact threshold was off by 1. Existing tests passed because they had the *same* off-by-one (`sigsAfterCompletion[1] = ['normalizes :name']` instead of `[0]`), so the bug in the code matched the bug in the test and they cancelled.

### Files changed

- `data/code-files.ts`: lowered every artifact threshold by 1 (`>= 0` for normalize, `>= 1` for enum + migration, `>= 2` for UsersController, `>= 3` for ProductsController). Comment block now explicitly documents the caller convention so future edits don't drift.
- `__tests__/Level15Callbacks.test.ts`:
  - "preview accumulates" test rekeyed from 1-indexed to 0-indexed (`{ 0: ['normalizes :name'], 1: ['enum :status'], 2: ['send_welcome_email(@user)'], 3: ['sync_to_accounting(@product.id)'] }`) so it agrees with the caller convention.
  - "preview shown while working on step k" leak test now passes `previewStep = k - 1` (was passing `k`, which was inconsistent with its own comment "= state after step k-1").
  - Added two new bug-reproduction tests: `after step 2 just completed, UsersController#create is visible` and `after step 3 just completed, ProductsController#mark_sold is visible`.

### Verification

- `bunx tsc --noEmit`: clean.
- `bunx biome check`: clean.
- `bun test`: 1637 pass / 0 fail (was 1635; +2 new artifact-visibility tests).
- `bun run build`: succeeds.

### Lesson

Two-bug-cancel: a buggy code threshold and a buggy test convention can mask each other indefinitely. The fix is to always anchor the test convention to a player-visible specification (in this case the CLAUDE.md `codePreviewStep` formula), not to whatever the existing code happens to do. Whenever a test labels its inputs with prose like "after step N completed" or "while working on step N," verify that label against the caller's actual semantics — not against the implementation under test.

This bug class probably exists in other levels' `code-files.ts`. Not sweeping today; flagging here so it can be picked up as a separate audit pass. Candidate test pattern for testing.md: every level with build-phase artifacts must assert that step-k's artifact is visible at `getCodeFiles('build', k)`, not at `getCodeFiles('build', k + 1)`.

### Rule additions (codify the case study so it gets caught next time)

User said this should have been caught earlier. Added a dedicated `audit-checklist.md` section "Step scope: on-concept + non-redundant (MANDATORY, SEPARATE STEP)" that walks the auditor through writing each step's one-sentence lesson and inspecting the list as a whole. Two failure shapes named with the L15 worked case:

1. **Off-concept step.** A step teaching a concept the level title doesn't name. Often shipped to feed a damage surface in the visualization. Drop the step (or broaden the level title honestly) — never carry an off-concept step to fill a slot.
2. **Duplicate-lesson step.** Two steps presenting the same structural choice between options. Drop one.

Plus a third related rule: **tail-wags-dog** — if the visualization needs N surfaces and the level only has M < N genuine on-concept lessons, cut surfaces, don't pad steps.

The rule additions live in:

- `.agents/rules/pedagogy.md` Build phase rules → two new sub-sections ("Every step must be on-concept with the level title" and "Two steps that teach the same structural choice are one step") + the L15 worked-failure case + the smell test "for each step, write down the one-sentence lesson it teaches."
- `.agents/skills/audit-level/audit-checklist.md` → new "Step scope: on-concept + non-redundant" section parallel to the feedback-leak scan and observe-code-preview-leak scan, with a worked table for L15 + bad/good examples + four explicit checklist items.
- `.agents/skills/audit-level/SKILL.md` Phase-2 quick-orientation line now names THREE separate mandatory passes (step scope scan + feedback leak scan + observe code preview leak scan), not just two.

### Lesson

Three earlier audits passed L15 at 4 steps because each step *individually* satisfied the option-quality and answer-leak rules. None of them asked the question that mattered: *"do these four steps teach four lessons, or two lessons twice plus one off-concept lesson?"* That question is now codified as a mandatory audit step. The trap was step-by-step thinking — the failure was visible only when looking at the step *set*. Same pattern as the feedback-leak scan and the observe-code-preview-leak scan: the failure shape demanded its own pass, separate from the structural option-quality review that was eating audit attention.
