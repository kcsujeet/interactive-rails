### Architecture: Single-Server Consolidation
**Before:** Two processes: Astro (port 4321) + Hono worker (port 8787), with Vite dev proxy forwarding `/api/*` requests. Deployed as Cloudflare Pages + Cloudflare Workers separately.

**After:** Single Astro server handles both pages and API. Hono app runs inside Astro via a catch-all API route. Deployed as one Cloudflare Workers application.

### Files Created
- `wrangler.jsonc`: Cloudflare config with D1 bindings, `main: "@astrojs/cloudflare/entrypoints/server"`
- `src/pages/api/[...path].ts`: Catch-all API route that delegates to Hono app, passing `env` from `cloudflare:workers`
- `src/server/`: All worker source code moved here (routes, middleware, services, repositories, utils, types, constants, errors, lib)

### Files Modified

### Key Decisions
- **Catch-all route pattern:** `src/pages/api/[...path].ts` imports the Hono app and calls `app.fetch(request, env)` where `env` comes from `import { env } from 'cloudflare:workers'`. This preserves all existing Hono routing, middleware, and error handling with zero refactoring.
- **Cloudflare Workers (not Pages):** Adapter v13 dropped Cloudflare Pages support. Deploy with `wrangler deploy`.
- **D1 bindings:** Accessed via `cloudflare:workers` module instead of `Astro.locals.runtime` (removed in v13).
- **Dev uses workerd:** Astro 6 + adapter v13 dev server uses real `workerd` runtime, matching production.

## Architecture Notes

### How the API Integration Works
```
Browser request to /api/pipeline/progress
    -> Astro server (workerd runtime)
    -> Catch-all route: src/pages/api/[...path].ts
    -> Hono app.fetch(request, env) with D1 bindings from cloudflare:workers
    -> Hono middleware (requestId, CORS, logging, rate limiting)
    -> Hono route handler (pipeline.ts)
    -> D1 database via c.env.DB
    -> JSON response back to browser
```
