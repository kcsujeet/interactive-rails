## Level 18 (Validation Contracts) revamp: static poster -> interactive three-phase

**What changed**

- **New file `ValidationGauntletFlow.tsx`**: a custom, state-driven visualization (mirrors L16's CallersFlow / L17's ConcernFlow pattern; no SMIL, so re-firing a scenario just replays the frame timers). A signup payload of five field chips (email_address, password, display_name, email_digest, role) flows into a validator. Before = the inline-checks service; after = a RegistrationContract with a Schema layer + Rules layer. Field chips light ok/bad/unchecked; a round-trip counter shows the cost.

  - **Intro (damage-first)**: replaced the static colored-code-blocks poster with the ValidationGauntletFlow in its broken state (service checked email, returned, never saw the other four fields) plus a "what the customer lives through" round-trip strip (Trip 1..4) and a Build-the-Fix button. Passes the smell test: a first-timer sees a signup that reports one error at a time and forces four resubmissions.
  - **Build**: unchanged (the 4 steps were already correct: install gem, schema approach, compose contract with `params(A, B, C)`, cross-field `rule` block).
  - **Reward (was static, now interactive)**: StressTestPanel + ValidationGauntletFlow with frame animation, exactly like L16. Four scenarios replay the intro damage stories with the fix: `all-errors-at-once` (schema flags all four fields in one 422), `malformed-payload` (numeric password -> clean 422 at the schema layer, never a 500), `creator-cross-field` (rules layer keys the failure to :role), plus `valid-signup` (201; the model still owns uniqueness). Left panel gains a Scenario heading + Allowed/Blocked counters.
  - Added `users_controller.rb` to the reward code snapshot (thin controller delegating to the service) so the `/users` stress path has a matching controller (reveal-consistency gate) and the loop is closed.

**Design provenance (design-level skill)**

- Form-axis level. Canonical purpose locked against dry-validation + dry-schema docs: the pattern validates external input at the boundary (dry-schema: "data structure and value types"; dry-validation: schema checks precede rule checks, "type safe"), fixing scattered imperative validation that reports one error at a time, has no type coercion, and buries/traps cross-field rules. The L16 before-state (inline `if @params[...].blank?` early-returns) exhibits exactly this.
- Matched L16/L17 (praised references), which are themselves `intro | build | reward` with a static intro + interactive StressTestPanel reward, NOT full observe-gating. So the revamp was "make the reward interactive + make the intro show the damage," not "add discovery probes."

**Learned**

- "Static intro" (Type 2) is not itself a defect in this curriculum: L16/L17 are Type-2 intros and were praised. The L18 defect was specifically the STATIC reward (a poster with a "Problems Solved" checklist) where the reference levels have an interactive StressTestPanel that lets the player fire the damage scenarios and watch the fix.
- State-driven frame snapshots (full `fields` array per frame + shallow-merged validator/edges) are far less bug-prone than SMIL animation and satisfy the "every scenario drives a distinct visible state" gate.
