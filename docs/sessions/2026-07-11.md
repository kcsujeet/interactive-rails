### Design decisions

- **The level's thesis:** the error tracker (L46) catches what BREAKS; observability is for what is silently wrong (slow, stuck, degraded) and raises nothing. Every probe is an incident where the tracker stays quiet, which resolves the cumulative-infrastructure conflict (post-L46, a plain outage WOULD page).
- **Smell test (written):** the player fires "Check the app is healthy after the 2am restart" and watches the monitor stay green on /up 200 while the worker node goes DEAD and customers stop receiving receipts for four hours. Reaction: "everything says fine and customers are getting nothing; I need eyes inside this thing." PASS.
- **Probe playthrough (written, per frame):** (1) hunt-slow-checkout: customers red CARTS ABANDONED -> logfile amber GREP "40 minutes in" -> logfile red UNQUERYABLE, customers SALES LOST. (2) trace-one-customer: ticket -> logs INTERLEAVED "six requests braided" -> NO THREAD, support cannot answer. (3) find-bottleneck: app 3.2s -> logfile ONE NUMBER -> app GUESSWORK, customers DAY 2. (4) check-worker-health: monitor GREEN 200 -> worker DEAD "jobs queue quietly, nothing raises" -> customers NO EMAILS while monitor still green. Four distinct shapes, test-enforced.
- **Topology (show-only-what-exists):** observe zones are all real: Customers, Rails App, production.log, Job Worker (Solid Queue, running since L36), Uptime Monitor pinging the real Rails 8 /up. The Trace Timeline node and its edge appear ONLY in reward (the build creates them); a test bans observe frames from touching the traces zone.
- **Reward:** 4 paired scenarios with "Same ..." replays plus one extra (pull everything one customer did today). check-worker-health is the blocked one: deep /up returns { database: true, job_worker: false } as 503 and the monitor pages at 2:09am; receipts flow by 2:20. Legend: "Caught: deep /up returned 503 and the monitor paged."
- **Build steps:** same 6-step chain, corrected: real gem versions; step 2 payload is request_id + user_id via Current.user (no tenant_id, no invented current_user helper); step 4 feedbacks no longer name use_all/service_name; step 5 reframed from "add the missing endpoint" to "deepen the built-in /up", with the wrong option being keep-the-default (which teaches the Rails 8 fact) and the correct answer backed by a fully shown HealthCheckService (SELECT 1 + SolidQueue::Process heartbeat freshness).
- **Content rewrite:** trigger/observation lead with the silent bad week; codeExample contrasts the interleaved log braid with the generated /up route; learning content keeps the percentile primer and flame-graph reading, drops Sidekiq/Redis/current_tenant/skip_before_action and the cross-service framing, and cites /up semantics to the Rails docs.
