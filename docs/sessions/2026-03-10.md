# Session Log: 2026-03-10

## Session 1: L25 StressTestPanel Response Lines

The StressTestPanel in L25's reward phase only showed one-line result logs (`200 GET /path actor`), while the ProbeTerminal in the observe phase showed multi-line colored response lines per probe. This made the stress test feel less informative than the probe terminal.

### Changes

**StressScenario type (`hooks/useStressTest.ts`):**
- Added optional `responseLines?: { text: string; color?: string }[]` field to `StressScenario` interface

**StressTestPanel (`components/levels/StressTestPanel.tsx`):**
- Renders `responseLines` below each result entry when present
- Color mapping: green -> emerald-400, red -> red-400, yellow -> yellow-400, default -> zinc-400
- Indented with `ml-8` to nest under the status line

**L25 component (`Level25NarrowFetching.tsx`):**
- Added `responseLines` to all 5 stress scenarios:
  - CSV Export: shows `User.pluck(:id, :email)`, 2.35 MB
  - Dropdown: shows `Category.pluck(:id, :name)`, 0.8 MB
  - API Response: shows `User.select(:id, :first_name, :last_name)`, 12.1 MB
  - Nightly Sync: shows `User.find_in_batches(batch_size: 1000)`, ~50 MB constant
  - Wide Fetch: shows `User.all`, 3.4 GB OOM killed (all red)
- Allowed scenarios show green lines (efficient), blocked shows red (wasteful)

**Audit skill (`.claude/skills/audit-level/SKILL.md`):**
- Added "StressTestPanel response lines (non-negotiable)" section before the button labels section
- Rule: if observe probes have responseLines, reward scenarios must too
- Observe lines show the problem (red), reward lines show the fix (green)
- Cross-phase parity: same endpoint, opposite story
- Exception: pure access-control levels (auth, CORS) where response is just allowed/forbidden
- 4-item checklist for response line quality

### Verification
- `bun run build` passes cleanly
- Backward compatible: levels without `responseLines` render unchanged

## Session 4: L26 Database Indexing Full Redesign

Audited L26 against the three-phase sequential flow standard. Found 13 issues including wrong visualization type (PipelineFlow instead of custom), missing discovery gating, missing animation locking, no progressive hints, no responseLines on stress scenarios, and stale startingPipeline in content.ts.

### Changes

**Level26Indexing.tsx (complete rewrite):**
- Replaced PipelineFlow-based visualization with custom "Query Plan Lanes" (Type 3)
- 3 horizontal lanes: email lookup, FK lookup, composite query, each with a ScanBar showing rows scanned
- ScanBar: red full bar = Seq Scan (all rows), green tiny bar = Index Scan (1-few rows)
- Added `useDiscoveryGating` with minRequired:4 (3 lane probes + schema inspector)
- Added animation locking: ProbeTerminal/StressTestPanel disabled during `isAnimating`
- Uses `ANIMATION_DURATION_MS` from `@/lib/animation`
- Progressive `<Alert variant="info">` hints after 2+ probes to guide clicking lanes/schema
- Added `responseLines` to all 5 stress scenarios (cross-phase parity with ProbeTerminal)
- Fixed code preview `getCodeFiles` duplicate file bug (was `if/if`, now `if/else if`)
- Fixed step 2 correct answer position (was last, now 2nd)
- Reward ScanBars show green (Index Scan) for allowed, red (Seq Scan) for blocked scenarios
- Clickable lanes with `?` indicators and `LANE_INSPECTOR_MAP` + `SCHEMA_INSPECTOR`

**content.ts (L26 definition):**
- Removed stale `startingPipeline` (nodes/connections array the component ignores)
- Removed stale `availableNodes`/`unlockedNodes` empty arrays
- Updated `trigger.description` to match new visualization ("Fire EXPLAIN probes to watch sequential scans...")
- Updated `hint.text` to reference query lanes and schema icon instead of "Database stage"

**Visualization redesign (ScanBar replaced with TableRowGrid):**
- User feedback: ScanBar progress bar "tells me nothing about indexing"
- Replaced ScanBar (simple progress bar) with TableRowGrid: 100 small row blocks in a 20-column grid
- Seq Scan animation: red wave sweeps through ALL blocks left-to-right, then match block(s) turn green
- Index Scan animation: IndexLookupCard shows B-tree entries with arrow to match, then only match block(s) turn green (no red wave)
- Added `IndexLookupCard`: shows sorted B-tree index entries (alice@example.com -> row #4231) with highlighted lookup
- Added `NoIndexBanner`: red banner showing "No index on {table}, Reading every row"
- Per-lane scan progress via `laneProgress` state (0 to GRID_SIZE) instead of binary `revealedLanes`
- Separate `rewardProgress`/`rewardDone` state for reward phase animations
- `runSeqScanAnimation`: 14 steps over ANIMATION_DURATION_MS, advancing blocks in chunks
- `runIndexScanAnimation`: short 400ms pause to show index card, then instant match reveal
- Grid match positions: email=[73] (1 match near end), fk=[12,37,54,71,89] (5 scattered), composite=[every other] (50% published)
- Grid legend below shows color meaning (scanned/matched/not touched)

### Architecture Notes

- L26 is a Type 3 (custom visualization) level because database indexing is about what happens INSIDE the database (scan strategy, row counts), not a missing MVC stage
- The TableRowGrid + IndexLookupCard is the core teaching tool: you SEE the database reading every row (red wave) vs jumping to the answer via index (green match only)
- Observe: NoIndexBanner + red sweep through all 100 blocks = "the database reads everything"
- Reward (index): IndexLookupCard shows B-tree + only match blocks turn green = "the index skips the scan"
- Reward (blocked): NoIndexBanner + red sweep again = "leftmost prefix rule violated, back to scanning"
