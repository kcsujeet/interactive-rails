# Session Log: 2026-07-17

## Terminal free-input: always-visible "Show the options" escape hatch

**What changed**

- `src/components/levels/terminal-input-matching.ts`: `shouldRevealOptions` now takes a second `manuallyRevealed` argument; options reveal either when the player asks or after `FREE_INPUT_MISS_LIMIT` (2) misses, whichever comes first.
- `src/components/levels/SimulatedTerminal.tsx`: the idle footer no longer says only "Type the command and press Enter." It now reads "Type the command and press Enter, or [Show the options]" with an always-visible outline Button that reveals the command choice buttons immediately. Removed the "One more miss reveals the options" line (redundant now that the button is always present). State resets per step via the existing `stepKey` remount in `TerminalChoiceStep`.
- `src/components/levels/__tests__/terminal-input-matching.test.ts`: new test "reveals immediately when the player asks, regardless of misses" plus updated signatures (TDD: written first, seen red, then implemented).

**Why**

Playtesting Level 1 as a first-time developer: the free-input terminal presents a blank prompt with zero affordance. The option buttons existed but were hidden behind two failed attempts, and the player was never told that failing reveals them. The only path forward was to flail on purpose without knowing flailing helps. Free-input-first stays (recall practice is the point of the design), but the recognition fallback is now discoverable from the start, so no learner is ever stuck.

**Checks**

tsc clean, biome clean, build 178 pages, 1828 unit tests pass (+1 new), 59 e2e pass (including the existing "two misses reveal options" loop test, which still holds since the manual reveal is additive).

**Learned**

The reveal state lives inside `SimulatedTerminal` and resets naturally on step change because `TerminalChoiceStep` keys the terminal block by `stepKey`. No prop threading needed; the escape hatch applies to every terminal step in every level automatically.
