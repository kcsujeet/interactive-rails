## L1 Environment: switch from asdf to mise

Replaced asdf with mise as the version manager taught in Level 1 (The
Environment). In 2026 mise is the recommended tool: Rust-based (faster), core
Ruby plugin built in (no `asdf plugin add ruby`), single-line shell activation,
and it still reads the standard `.tool-versions` format so the rest of the
level's mental model is unchanged.

  - `conceptExplanation` rewritten to highlight mise's Rust speed, built-in
    Ruby support, and shell activation model.
  - `railsCodeExample` now uses `brew install mise`, `mise activate zsh`,
    and `mise install`.
  - `furtherReading` link swapped for https://mise.jdx.dev/.

## L1 Environment: switch `.tool-versions` -> `.mise.toml`

Follow-up on the mise migration: the config step now teaches the mise-native
TOML format instead of the asdf-compat `.tool-versions`. Reasoning: if we're
already teaching mise, teach the format mise's own docs recommend. `.mise.toml`
is also what later levels would extend with env/tasks, so anchoring on it in
L1 makes the progression coherent.

### What changed

- `Level1Environment.tsx`
  - Step 3 renamed from `tool-versions` -> `mise-toml`, title "Configure
    .mise.toml".
  - Wrong options exercise three realistic TOML/config mistakes:
    - YAML colon syntax (`ruby: "3.3.6"`)
    - asdf-style space-separated (`ruby 3.3.6`) inside a `.toml` file
    - bare TOML key with no section (`ruby = "3.3.6"`)
  - Correct option uses section syntax. Label uses `>` as an inline separator
    (`[tools] > ruby = "3.3.6"`) because button labels render single-line;
    the `command` uses `printf '[tools]\nruby = "3.3.6"\n'` to produce the
    real multi-line file contents.
  - Step 4 description and `brew install ruby` feedback now reference
    `.mise.toml` instead of `.tool-versions`.
  - `getCodeFiles` step-3 preview switched to a `toml` file showing
    `[tools]\nruby = "3.3.6"`.

- `level-1-environment.ts`
  - `trigger.description` generalised ("project config file") so the filename
    isn't spoiled before the player hits step 3.
  - `codeExample` comment block no longer names `.tool-versions`.
  - `conceptExplanation` "The .mise.toml file" section replaces the old
    `.tool-versions` section; adds a note that the same file can hold env
    vars and tasks.
  - `railsCodeExample` uses a heredoc to write `[tools]\nruby = "3.3.6"` into
    `.mise.toml`.
  - `commonMistakes[1]` updated to the real TOML pitfall ("bare key without
    section header") instead of the old hyphen/colon pitfall.
  - `whenToUse` and `hint.text` updated to reference `.mise.toml`.

## LevelHeader: two-step Submit -> Complete flow

The Submit button had three label states (`Submit`, `Complete Level`,
`Completing...`) but `handleSubmit` validated *and* completed in the same
click, so the green `Complete Level` state was unreachable and a successful
validation jumped straight to the redirect. Players never got a chance to see
the success banner or confirm.

### What changed

- `src/components/levels/LevelHeader.tsx`
  - `handleSubmit` now short-circuits to `onComplete()` when `lastResult?.valid`
    is already true (i.e. the button is already in the `Complete Level`
    state). Otherwise it just validates and surfaces feedback.
  - Net effect: first click validates, second click completes. No config/UI
    changes elsewhere were needed; the button label already reflected this
    intended flow.
  - Cleaned up pre-existing warnings in the same file while touching it
    (etiquette rule: own every error in files you touch):
    - Removed unused `useMemo` import.
    - Removed unused `buildUnifiedProjectAtStep` import.
    - Removed unused `completedLevels` and `currentLevelId` props (no caller
      was passing them).
