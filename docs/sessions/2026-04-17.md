## Observe-phase gating: all probes required (non-negotiable)

Player feedback: "Build the Fix" showed up before all probes were fired.
Root cause was that two of L42's probes each unlocked two discoveries, so
firing two probes could clear a `minRequired: 3` gate.

Promoted "every probe must fire" to a non-negotiable project rule.

### CLAUDE.md change

The observe-phase bullet now reads: the button appears **only when every
probe has been fired and every discovery unlocked**. Two enforcement
clauses:

1. `useDiscoveryGating(DISCOVERY_DEFS, { minRequired: DISCOVERY_DEFS.length })`
   — always require all.
2. `PROBE_DISCOVERY_MAP` must be strictly 1:1: each probe unlocks exactly
   one distinct discovery; each discovery is unlocked by exactly one
   probe. A discovery may never be reused across probes.

### L42 changes

- `Level42Deployment.tsx:31`: `minRequired: 3` -> `minRequired: DISCOVERY_DEFS.length`.
- `ObservePhase.tsx`: hardcoded `minRequired={3}` -> `discoveryGating.discoveries.length`.
- `data/probes.ts`: trimmed rollback + two-servers mappings to one
  discovery each (already done earlier for the 1:1 fix).
- New test `probe-to-discovery mapping is strictly 1:1` encodes the rule
  so any regression fails CI.

## L42 design pass: all five gaps

Applied the `/design-level` skill to L42 and executed every gap identified
in the resulting design doc. Five substantive changes.

### Gap 1: probe + scenario stories

Every probe (5) and every reward scenario (5) now has a `story: string[]`
with 3–6 narrative bullet points. Shown in the info modal on click; gives
the player the "why" behind each action without cluttering the button
label.

### Gap 2: live pipeline during the build phase

`BuildPhase.tsx` now renders `PipelineFlow` above the step content. A new
`buildStagesFor(completedStep)` in `data/pipeline-stages.ts` materializes
one stage at a time as the player advances:

- Step 0 (`bundle add kamal`): Laptop sublabel updates
- Step 1 (`kamal init`): Registry + Proxy + Container flip ghost → default stubs
- Step 2 (deploy.yml): Registry "awaiting image", Proxy "/up health check"
- Step 3 (secrets): Registry → active
- Step 4 (`kamal setup`): all stages active, server healthy

The build phase is no longer a step-only screen; the pipeline is visible
and evolves with every action the player takes.

### Gap 3: v1/v2 rotation in the reward phase

Reward scenarios now play multi-frame sequences that explicitly model
container rotation. The Container node sublabel shows the live state:

- `deploy-ok`: "v1 live | v2 booting" → "v2 healthy | v1 stopped"
- `deploy-broken-health`: "v1 live | v2 unhealthy" (never shifts)
- `rollback`: "v2 live (bad)" → "v1 container reused"
- `fleet-deploy`: "prod1: v1→v2 | prod2: v1" → "prod1: v2 | prod2: v1→v2" → "prod1: v2 | prod2: v2"
- `broken-push`: v2 stays healthy, laptop node shows "docker build FAILED"

### Gap 4: scenario label rename

`broken-push` label renamed from "Push an image that fails to build" to
"Deploy with a broken build" so it mirrors the `git-pull` probe label
"Deploy by ssh + git pull". New `probe-scenario label mirroring` test
encodes which probe pairs with which scenario and which thematic words
must appear in both labels; this test would now catch any future drift.

### Gap 5: multi-frame animation

Replaced single-frame-per-probe reactivity with true frame playback.

- New type `PipelineFrame { stages; durationMs }` in `data/pipeline-stages.ts`.
- `PROBE_FRAMES` and `SCENARIO_FRAMES` records: each probe/scenario has a
  3-frame sequence (setup → hit → fallout), each frame shown for 900–1600ms.
- New hook `usePipelineFrames()` owns the playback state machine: takes a
  frame array, walks through it via chained `setTimeout`s, exposes
  `currentFrame` and `isPlaying`. Clears its timers on unmount and when a
  new sequence starts.
- `ObservePhase` and `RewardPhase` consume the hook. `ProbeTerminal` and
  `StressTestPanel` both receive `disabled={isPlaying}` so the player
  cannot trigger a second animation while one is running.

### Test additions

- Every probe has a 3–6 bullet story with each bullet > 20 chars
- Every scenario has a 3–6 bullet story
- `Probe-scenario label mirroring` describe block: asserts every probe has
  a declared scenario pair, and that each pair shares a thematic word in
  both labels (e.g., "Deploy", "Roll back", "Ship a release")
