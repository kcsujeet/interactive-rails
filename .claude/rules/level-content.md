# Level Content

- **Never reveal answers in any player-facing text.** The trigger description, problem observation, **problem goal**, learningContent goal, left panel scenario text, step descriptions, hint text, and **wrong-option feedback** must never name the correct gem, method, class, or command that the player will choose in a build step. Describe the requirements, constraints, and outcomes instead, so the player discovers the answer themselves. The goal says WHAT to achieve ("implement soft deletes with an audit trail"), not HOW ("install discard gem and PaperTrail").

- **Build for first-time aspiring developers.** This project is built for Rails newcomers, not Rails veterans. **Every piece of player-visible content, text and visual alike**, must be understandable to a smart beginner. Existing text and visuals with jargon are grandfathered, but **do not add more, and prefer to plain-rewrite when you touch a section.**

  - **For text copy** (concept explanations, step descriptions, hints, scenario text, wrong-option feedback, common mistakes, stage inspector descriptions, terminal output, probe response lines, code-preview comments): avoid jargon stacks ("thread-local per-request", "`ActiveSupport::CurrentAttributes` subclass", "constant-time comparison", "delegate to session"). Lead with **why this matters** ("after login, your code needs to know who the user is"), then **how Rails does it** ("Rails stores it in `Current.user`"). Use concrete framings for confusing patterns ("how does anyone log in for the first time if logging in requires being logged in?"). Name the API the player will see in code (`Current.user`, `allow_unauthenticated_access`) but never assume they understand the metaprogramming machinery underneath. Plain phrasing beats technical accuracy: "block everyone by default, then carve out specific exceptions" is clearer than "deny by default, allow explicitly."

  - **For visualizations** (pipeline flows, query zones, probe responses, stress-test outputs, animation frames, badges, sublabels, stage inspector cards, custom flow diagrams): the visual must convey the concept the level is teaching, with no extra surface area. Label nodes with what they ARE in plain English ("Database", "Cache Store", "Login Endpoint", "Background Job"), not with internal class names or framework jargon ("ActionDispatch::Routing", "Rack::Builder", "ApplicationCurrentAttributes"). Probe and stress responses should show numbers and short prose a beginner can read ("101 queries, 850ms"), not raw EXPLAIN output or stack traces unless the *level itself* is teaching how to read them. Color cues must be intuitive and consistent: red = problem / blocked, green = working / allowed, amber = warning. Do not introduce visual elements (zones, badges, sublabels, animation states) for concepts the level has not taught yet. One probe should reveal **one** distinct piece of the problem, not five at once. If a node, badge, or animation needs a glossary to understand, it is teaching the wrong audience.
